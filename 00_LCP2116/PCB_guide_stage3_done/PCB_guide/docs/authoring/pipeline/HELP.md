# HELP — быстрые команды для работы с ассистентом

Этот файл — **короткая карта действий** для автора, который не хочет читать длинные README перед стартом.

Если автор загрузил актуальный ZIP-архив репозитория и написал в чат только слово:

> help

ассистент должен ответить **коротким help-меню**, а не требовать от автора помнить структуру пайплайна.

---

## один. Что делает команда `help`

Если автор пишет только:

> help

ассистент обязан:

* подтвердить, что архив репозитория загружен;
* кратко объяснить, что это интерактивный пайплайн;
* перечислить доступные команды / сценарии;
* объяснить, как начать с любой стадии;
* объяснить, как восстановиться после потери контекста;
* подсказать, где лежат длинные инструкции.

Ответ должен быть **коротким, практическим и пригодным для копирования**.

---

## два. Минимальное help-меню, которое должен выдавать ассистент

Ниже — канонический состав краткого help-ответа.

### Базовые команды

* `help` — показать краткую карту действий и доступные команды.
* `show readme` — кратко показать, что делает весь репозиторий и где что лежит.
* `show pipeline` — кратко показать стадии и общий маршрут работы.
* `detect stage` — определить текущую стадию по артефактам в архиве.

### Запуск стадий

* `start stage 0`
* `start stage 1`
* `start stage 2`
* `start stage 3`
* `start stage 4`
* `start stage 5`

### Повторный вход

* `re-entry stage X` — восстановить ход по конкретной стадии.
* `author action done` — сообщить, что автор выполнил handoff и загрузил новый архив.

### Навигация по документам

* `where is start here`
* `where is help`
* `where is pipeline`
* `where is reentry prompts`

---

## три. Что именно должен говорить ассистент на `help`

Рекомендуемый формат ответа:

### Краткое вступление

* это интерактивный пайплайн;
* вы загружаете актуальный архив;
* ассистент ведёт вас по стадиям и выдает handoff.

### Команды, которые можно копировать сразу

```text
help
show readme
show pipeline
detect stage
start stage 0
start stage 1
start stage 2
start stage 3
start stage 4
start stage 5
re-entry stage X
author action done
```

### Короткая расшифровка

* `detect stage` — если вы не уверены, где остановились;
* `start stage X` — если стадия известна;
* `author action done` — если вы сделали свою часть и загрузили новый архив;
* `show pipeline` — если хотите увидеть краткую карту стадий.

### Где лежат длинные инструкции

* `docs/authoring/pipeline/START_HERE.md`
* `docs/authoring/pipeline/HELP.md`
* `docs/authoring/pipeline/REENTRY_PROMPTS.md`
* `docs/authoring/pipeline/PIPELINE.md`

## четыре. Канонические короткие сообщения автора

### Если автор только начал

> help

### Если автор хочет стартовать с конкретной стадии

> start stage 2

### Если автор не помнит стадию

> detect stage

### Если автор выполнил свою часть работы

> author action done

### Если открыт новый чат и контекст потерян

> re-entry stage 3

---

## пять. Правило робастности

Ассистент должен интерпретировать эти короткие команды **не как буквальный API**, а как дружелюбные команды автора.

Примеры допустимых эквивалентов:

* `help`
* `помощь`
* `покажи help`
* `что дальше`
* `на какой мы стадии`
* `определи стадию`
* `начинай stage 3`
* `возврат в stage 4`

Если команда неоднозначна, ассистент сначала должен попробовать восстановить состояние по архиву, а не сразу требовать длинного пояснения от автора.
