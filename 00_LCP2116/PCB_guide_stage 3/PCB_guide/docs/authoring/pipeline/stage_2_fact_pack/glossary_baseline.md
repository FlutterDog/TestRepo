# Glossary baseline — Stage 2 (`glossary.md` seed)

> ℹ️ **Назначение документа**
> Это **канонический baseline-словарь** для Stage 2.
>
> Он не является готовым project artifact для всех плат.
> Его задача — дать **стартовую базу терминов, единиц и правил**, из которой на Stage 2 создаётся проектный:
>
> * `docs/authoring/artifacts/stage_2_fact_pack/glossary.md`

---

## Purpose

Этот baseline нужен для того, чтобы:

* не писать `glossary.md` с нуля на каждом проекте;
* удерживать единые статусы, единицы и базовые термины между проектами;
* не превращать project glossary в случайный словарь «по памяти»;
* сохранить место для project-specific дополнений по конкретной плате.

❗ Важное правило:
этот документ — **стартовая база**, а не финальный glossary для всех случаев.

Project-specific `glossary.md` на Stage 2 должен:

* наследовать правила этого baseline;
* включать только реально используемые в проекте термины и соглашения;
* при необходимости дополняться board-specific терминами, доменами и локальными naming rules.

---

## 1. Статусы утверждений

* **FACT** — подтверждено источником.
  В authoring-артефактах должен быть указан якорь подтверждения: файл / лист / страница / RefDes / сеть / иной точный указатель.

* **ASSUMPTION** — допущение.
  Нужно зафиксировать причину, риск и что требуется для подтверждения.

* **UNKNOWN** — неизвестно / не подтверждено.
  Нужно указать, какого источника или данных не хватает.

### Правило применения

Любое значимое утверждение в authoring-зоне должно быть либо:

* подтверждено как `FACT`;
* явно помечено как `ASSUMPTION`;
* явно помечено как `UNKNOWN`.

❗ В релизном `docs/release/board_guide.md` эти статусы **не показываются как служебные метки**.
Они живут в authoring-артефактах:

* `file_map.md`
* `fact_pack.md`
* `open_items.md`
* `qc_report.md`

## 2. Иерархия Source of Truth

При конфликте данных использовать такой приоритет:

1. **Native CAD project** — первично для электрической логики, связности, RefDes, имён сетей и назначения узлов.
   Примеры: `SchDoc / PcbDoc` для Altium или эквивалентные native-файлы другой CAD-системы.

2. **PDF outputs схемы / платы** — использовать как подтверждение, если они соответствуют ревизии и явно относятся к актуальному комплекту.

3. **Gerber / NC Drill** — первично для геометрии, слоёв, отверстий и части manufacturing-фактов, если native CAD недоступен или требует верификации.

4. **BOM export** — первично для состава, DNP/DNI, part numbers, альтернатив и части supply-side логики, если это подтверждено содержимым файла.

5. **Stackup report / fabrication export** — первично для структуры слоёв, толщин, материалов и controlled impedance, если документ относится к актуальной ревизии.

6. **Proof materials / written notes / correspondence** — только как вспомогательный уровень.
   Такие источники нельзя бездумно поднимать до уровня основного доказательства, если конфликтуют более сильные инженерные источники.

### Правило трассируемости

Каждый значимый факт должен быть трассируем через:

* `file_map.md`;
* и/или `fact_pack.md`;
* и/или прямой якорь в `sources/*`.

❗ Это не означает, что в релизной прозе нужно показывать источник в явном виде.
Явная трассируемость обязательна для **authoring**, а не для чистого release-текста.

## 3. Единицы измерения и формат записи

* Напряжение: **V**
* Ток: **A**
* Сопротивление: **Ω**
* Ёмкость: **F**
* Частота: **Hz**
* Температура: **°C**
* Размеры: **mm** по умолчанию
* Толщина меди: **oz** и/или **µm**
* Импеданс: **Ω**
* Дифференциальный импеданс: **Zdiff**

Префиксы:

* `m`, `µ`, `n`, `p`, `k`, `M`, `G`

Десятичный разделитель:

* **точка** — например `3.3 V`

### Правило консистентности

В рамках одного проекта не смешивать альтернативные стили записи без причины.
Например:

* не чередовать `3,3 V` и `3.3 V`;
* не чередовать `миллиметры` и `mm` внутри технических мест без необходимости;
* не менять обозначение домена питания между разделами.

## 4. Сокращения — baseline minimum

В project glossary включать только те сокращения, которые реально используются в текущем проекте.

Базовый минимальный набор:

* **PCB** — печатная плата
* **Schematic** — электрическая схема
* **BOM** — ведомость состава / элементная база
* **DFM** — Design for Manufacturability
* **DFT** — Design for Test
* **Bring-up** — первичный запуск платы
* **TP** — test point / контрольная точка
* **ESD / EFT / Surge** — электростатика / быстрые импульсные помехи / грозовые импульсы
* **TVS** — защитный супрессор
* **PPTC / PTC** — самовосстанавливающийся предохранитель
* **LDO** — линейный стабилизатор
* **Buck / Boost** — понижающий / повышающий DC/DC
* **GND / AGND / DGND / PGND / Chassis** — земля / домены земли
* **PE** — защитное заземление
* **EMI / EMC** — электромагнитные помехи / совместимость
* **DNP / DNI** — не устанавливать
* **RefDes** — позиционное обозначение
* **Net** — имя цепи

### CAD-specific additions

Если проект реально использует Altium-термины, допустимо добавить:

* **SchDoc / PcbDoc** — native-файлы Altium

Если проект использует другую CAD-систему, в project glossary надо добавить её native-термины вместо или вместе с Altium-терминами.

## 5. Соглашения по именам и ссылкам

### Позиционные обозначения

* использовать **как в проекте**;
* не вводить собственные переименования RefDes в тексте.

### Имена сетей и доменов питания

* использовать **как в проекте**;
* не нормализовать вручную `+3V3`, `3V3`, `VIN_24V` и подобные сущности, если в SoT они заданы конкретным образом.

### Где показывать «где смотреть»

Явные source anchors обязательны в authoring-артефактах, например:

* `Sch: <sheet/file> / RefDes: U3, R12`
* `PCB: <layer/area> / координаты, если доступны и полезны`
* `PDF: <file> / p. N`
* `Gerber: <file/layer>`
* `BOM: <file/sheet/column>`

❗ В релизном тексте такие ссылки допустимы только если они реально улучшают инженерную полезность раздела.
Они не должны превращать release в process-report.

## 6. Термины по электрическим узлам

* **Power domain** — домен питания: уровень напряжения и группа потребителей / область применения
* **Protection chain** — цепочка защиты: например `PPTC → TVS → filter → regulator`
* **Bias / Fail-safe (RS-485)** — смещение линии для устойчивого состояния в покое
* **Termination** — терминатор на линии или дифференциальной паре
* **Return path** — путь обратного тока
* **Stitching vias** — сшивающие переходные отверстия
* **Tenting** — закрытие via маской

---

## 7. Термины по PCB / stackup

* **Stackup** — стек слоёв: материалы, толщины, медь, препреги
* **Core / Prepreg** — сердцевина / препрег
* **Copper weight** — толщина меди, обычно в `oz`
* **Controlled impedance** — контролируемый импеданс трасс

---

## 8. Сигнальные интерфейсы — только physical layer

Добавлять в project glossary только те интерфейсы, которые реально есть на плате.

Базовые примеры:

* **RS-485** — дифференциальная линия: терминатор, TVS, bias, трассировка
* **Ethernet** — пары, трансформатор, разъём, экран, импеданс
* **I²C / SPI / UART** — уровни, подтяжки, защита линий

❗ Здесь фиксируются только electrical / PCB-level аспекты.
Протоколы, прошивка и system-level behavior не входят в glossary baseline.

---

## 9. Мини-чеклист консистентности терминов

* Один термин — одно значение.
* Не смешивать без причины `земля`, `общий`, `минус`, если речь идёт о разных сущностях.
* `AGND / DGND / PGND` упоминать только если они реально разведены или явно выделены в SoT.
* `Shield / chassis / PE / GND` всегда различать и явно указывать, как сущность подключена.
* Не вводить project-local жаргон без явного определения.

---

## 10. Что должно быть добавлено в project glossary

Project-specific `docs/authoring/artifacts/stage_2_fact_pack/glossary.md` должен при необходимости дополнительно включать:

* реальные имена power domains текущей платы;
* board-specific сокращения и внутренние названия узлов;
* соглашения по локальным именам интерфейсов и разъёмов;
* допустимые и недопустимые синонимы для конкретного проекта;
* CAD-specific термины, если проект не Altium-based;
* любые термины, которые реально появляются в `fact_pack.md`, `outline.md`, `draft_v0.md` и будущем релизе.

### Правило отбора

В project glossary не нужно копировать весь baseline механически.
Нужно оставить:

* всё, что реально используется;
* всё, что нужно для консистентности проекта;
* всё, что предотвращает drift терминов между Stage 2–5.
