# Stage 4 — Draft (`draft_v0.md`)

## Purpose

Stage four создаёт **первый полный объясняющий черновик**.

Это уже не skeleton и не тезисы. Это уже читаемый инженерный документ, который:

* опирается на outline;
* использует facts, capabilities, design intent и author notes;
* объясняет плату как реализованную платформу;
* не скатывается ни в сухую инвентаризацию, ни в маркетинговую подачу.

---

### Обязательные

* `outline.md`
* `fact_pack.md`
* `glossary.md`

* `reference_sources.md`

---

## Procedure

### Шаг ноль. Проверить, завершён ли author pass после Stage three

Перед сильным Draft ассистент обязан проверить:

* заполнен ли `author_notes.md` или пользователь осознанно отказался от него;
* подготовлены ли planned author-owned figures / tables / excerpts или пользователь осознанно идёт дальше без них;
* хватает ли reference sources для характеристик и возможностей, которые ожидаются в тексте.

Если этих слоёв не хватает, ассистент не должен тихо притворяться, что Stage four полностью обеспечен. Он обязан явно сообщить пользователю, чего не хватает.

### Шаг один. Писать строго по outline

### Шаг два. Использовать все разрешённые слои данных

На Stage four разрешено и нужно использовать:

* implemented facts;
* component capability facts;
* board-supported capabilities;
* design intent;
* author notes.

### Шаг три. Для каждого крупного узла писать не менее пяти смысловых слоёв

* что это за узел;
* как реализован;
* какие характеристики важны;
* что это даёт пользователю или интегратору;
* какие ограничения и инженерные последствия у модификаций.

### Шаг четыре. Не переписывать даташиты целиком

### Шаг пять. Использовать author notes как усилитель текста

### Шаг шесть. Корректно обходиться с figures и tables

* Если asset уже готов, использовать рабочую ссылку / место вставки по outline.
* Если asset запланирован, но ещё не готов, Draft должен сохранять для него логическое место и не терять его смысл.
* Для planned, но отсутствующих assets ассистент должен вставлять **draft placeholder block** прямо в текст Draft.
* Placeholder block должен содержать:
  * HTML comment с Figure ID, type, owner и status;
  * markdown image-link или table marker с целевым путём в `IMG/`;
  * короткую видимую note-строку, что именно ожидается подготовить.
* Если отсутствие asset ухудшает читаемость главы, это должно быть явно отмечено для пользователя как действие перед Stage five.

Канонический вид figure placeholder:

`<!-- FIGURE_PLACEHOLDER: F-ноль один | type=diagram | owner=author | status=to_prepare -->`

`![Структурная схема платы и основных функциональных блоков.](IMG/block_diagram.png)`

`_Запланированная иллюстрация: структурная схема платы и основных функциональных блоков. Подготовить файл `IMG/block_diagram.png`._`

Такой блок позволяет автору сразу видеть ожидаемое место, имя файла и назначение figure.

## Что ассистент должен сказать в конце Stage four

Ассистент обязан явно сообщить:

* получился ли Draft полноценным или ограниченным;
* какие author-owned assets всё ещё не хватает для сильного релиза;
* можно ли уже идти на Stage five или сначала нужен author pass.

## DoD

Stage four завершён, если:

* `draft_v0.md` покрывает весь outline;
* текст уже читается как инженерный документ;
* в разделах есть не только реализация, но и характеристики, возможности и ограничения;
* нет сухих глав, сведённых к трём предложениям;
* нет выдуманных чисел и неподтверждённых обещаний;
* author notes, если были, использованы по назначению;
* по figures и tables понятно, что уже готово, а что ещё нужно подготовить до финального релиза.


---

## Как использовать style exemplars на Stage four

Если в проекте есть документы в `docs/authoring/pipeline/style_references/`, ассистент может использовать их только для:

* выбора плотности текста;
* выбора места для figures, tables и diagrams;
* выбора типа explanatory inserts.

Ассистент не должен переносить из них factual claims по вашей плате или по применённым компонентам.
