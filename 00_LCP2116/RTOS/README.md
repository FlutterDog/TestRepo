## 1. Общие сведения о платформе Lorentz

Lorentz представляет собой свободно программируемую модульную систему управления, предназначенную для построения распределённых систем автоматизации, сбора данных, контроля технологического оборудования и управления исполнительными устройствами.

Центральным элементом системы является контроллер **LCP2116**. Он выполняет прикладную программу, обслуживает внутреннюю модульную шину, взаимодействует с внешними приборами и предоставляет сетевые и сервисные интерфейсы.

В состав линейки входят функциональные модули различного назначения, в том числе:

- модули дискретных входов LDI;
- модули дискретных выходов LDO;
- модули аналоговых входов LAI;
- модули контроля коммутационного ресурса LCT;
- центральный контроллер LCP2116.

Конкретный состав системы определяется проектом. Модули устанавливаются в требуемом порядке, а центральный контроллер выполняет их последовательный опрос и предоставляет данные прикладному программному обеспечению.

LCP2116 построен на микроконтроллере ATSAM3X8E с ядром ARM Cortex-M3. Базовая аппаратная платформа предоставляет:

- внутреннюю шину X2X для модулей системы;
- четыре пользовательских последовательных порта S1–S4;
- универсальный последовательный интерфейс HMI с физическим уровнем RS-485;
- два независимых Ethernet-интерфейса на W5500;
- USB CDC для служебной консоли и машинного конфигурационного протокола;
- microSD как опциональный носитель совместимого конфигурационного комплекта;
- RTC и резервную батарею;
- аппаратный watchdog;
- внешние последовательные каналы на SC16IS7xx;
- аппаратные ресурсы для дальнейшего расширения прикладного программного обеспечения.

> 🟦 **Информация**
>
> Название HMI сохранено как обозначение физического интерфейса платы и не ограничивает его применение. Разработчик может использовать этот RS-485-порт для панели оператора, внешнего прибора, сервисного оборудования или собственного протокола.

> 🟦 **Информация**
>
> Данное руководство описывает базовую прошивку LCP2116 версии 1.02.0 и порядок её расширения. Специализированная логика конкретного объекта реализуется разработчиком поверх описанной программной архитектуры.

Конкретные карты регистров, алгоритмы и эксплуатационные особенности отдельных функциональных модулей должны находиться в документации соответствующих модулей. Настоящий README описывает центральный контроллер, общую X2X-архитектуру и точки расширения.

## 2. Назначение и границы базовой прошивки LCP2116

`LCP Basic Diagnostic Firmware` является исходной программной платформой центрального контроллера LCP2116. Прошивка обеспечивает запуск оборудования, базовый обмен данными, диагностику и набор готовых примеров для разработки прикладного проекта.

В версии 1.02.0 реализованы:

- FreeRTOS;
- одна основная прикладная задача LCP;
- неблокирующая модель обслуживания подсистем;
- X2X master на UART0;
- статический каталог и реестр X2X-модулей;
- четыре независимых Modbus RTU master на портах S1–S4;
- неблокирующий диагностический echo-test универсального порта HMI;
- два независимых Modbus TCP server на ETH1 и ETH2;
- нормализованная платформенная конфигурация schema-v1;
- надёжное A/B-хранилище конфигурации во внутренней Flash ATSAM3X8E;
- встроенные runtime defaults для запуска нового контроллера;
- бинарный USB CDC-протокол чтения, проверки и записи конфигурации;
- опциональный строгий импорт совместимого TXT-комплекта с microSD;
- служебная консоль USB CDC;
- reference Python utility для конфигурирования без SD-карты;
- RTC, резервная батарея и аппаратный watchdog;
- диагностика памяти, UART, microSD, конфигурации, X2X, FieldSensor и Ethernet;
- примеры добавления внешнего прибора, нового X2X-модуля и новой команды консоли.

Прошивка организована как набор независимых уровней. Универсальные реализации протоколов отделены от аппаратной привязки и прикладных сервисов. Это позволяет добавлять новые типы оборудования без изменения базового Modbus engine или низкоуровневого HAL.

Базовая прошивка не содержит:

- прикладного алгоритма конкретной технологической установки;
- обязательной структуры пользовательского проекта;
- обязательного формата пользовательских рецептов, уставок или карт сигналов;
- универсального USB–RS-485 bridge;
- отдельной команды `test start`;
- автоматизированного стенда ОТК;
- LabVIEW-приложения;
- производственной логики PASS/FAIL;
- PNX Studio.

> 🟦 **Информация**
>
> LCP Basic одновременно является рабочей диагностической программой контроллера и исходной архитектурой для дальнейшей разработки. TXT, JSON, Python и microSD являются доступными способами конфигурирования или примерами интеграции, но не навязываются прикладному проекту.

## 3. Быстрый старт

### 3.1 Запуск нового контроллера без microSD

Минимальная последовательность:

1. Открыть проект Microchip Studio:

   ```text
   00_LCP2116/RTOS/LCP_Basic.cppproj
   ```

2. Выбрать конфигурацию `Release`.
3. Собрать проект без ошибок и предупреждений исходного кода.
4. Прошить LCP2116.
5. Подключить USB CDC.
6. Открыть терминал и выполнить:

   ```text
   version
   help
   status
   rtos
   config status
   field
   x2x
   eth
   sc16is
   sd
   time
   watchdog
   ```

Ожидаемая версия:

```text
version = 1.02.0
stage = Release 1.02.0
target = ATSAM3X8E
```

При пустой конфигурационной Flash и отсутствии корректного SD-комплекта контроллер через ограниченное начальное ожидание автоматически запускается на встроенных runtime defaults:

```text
source = defaults
slot = none
sequence = 0
```

microSD для обычного запуска не требуется.

### 3.2 Ручная работа без Python

Через текстовую консоль можно:

- проверить текущий источник и оба A/B-слота;
- вывести активную конфигурацию в каноническом TXT-виде;
- проверить полный комплект на microSD без записи;
- сравнить microSD с активной конфигурацией;
- импортировать проверенный комплект с microSD;
- выполнить rollback;
- инвалидировать оба слота и вернуться к runtime defaults;
- проверить основные аппаратные подсистемы.

Основные команды:

```text
config status
config show all
config scan sd
config compare sd
config import sd
config rollback
config erase confirm
config defaults
```

Команда `config defaults` только выводит встроенные значения. Она не активирует и не записывает их во Flash. Для ручной установки произвольной постоянной конфигурации без Python используется полный совместимый комплект на microSD и команда `config import sd`.

### 3.3 Конфигурирование через USB с reference utility

На компьютере должны быть установлены Python 3.9+ и `pyserial`:

```powershell
python -m pip install pyserial
```

Проверка связи:

```powershell
python .\tools\lcp_config_usb.py info --port COM10
```

Чтение активной конфигурации:

```powershell
python .\tools\lcp_config_usb.py read `
    --port COM10 `
    --output .\test_output\active_config.json
```

Запись JSON-модели:

```powershell
python .\tools\lcp_config_usb.py write `
    --port COM10 `
    --input .\my_config.json
```

Импорт совместимого TXT-каталога и запись:

```powershell
python .\tools\lcp_config_usb.py write-txt `
    --port COM10 `
    --directory D:\LCP_Config_USB_Test
```

Номер COM-порта и путь к конфигурации зависят от конкретного компьютера.

## 4. Архитектура программного обеспечения

### 4.1 Общая структура

Программное обеспечение разделено на уровни с явно определённой ответственностью:

```text
main.cpp
    точка входа и запуск FreeRTOS

app/
    прикладные сервисы, runtime-состояния, конфигурация и диагностика

board/
    привязка логических интерфейсов к ресурсам платы LCP2116

hal/
    низкоуровневая работа ATSAM3X8E, SC16IS7xx, W5500 и Flash

protocol/
    универсальные реализации Modbus RTU и Modbus TCP

platform/
    Serial, SPI, GPIO, millis и другие platform-функции

libs/
    независимые повторно используемые библиотеки, USB, CRC32 и SD storage

middleware/
    FreeRTOS Kernel

tools/
    reference Python utility конфигурирования через USB

sd_card/
    пример совместимого канонического TXT-комплекта

IMG/
    изображения, используемые данным README
```

В проекте намеренно нет отдельного каталога эксплуатационной документации: основное объединённое руководство находится в настоящем `README.md`.

### 4.2 Назначение уровней

```text
HAL       -> регистры MCU и физические аппаратные операции
board     -> разводка и ресурсы конкретной платы
protocol  -> универсальная реализация протокола
service   -> применение протокола для конкретной функции
status    -> отображение состояния и обработка сервисных команд
```

Пример прохождения данных FieldSensor:

```text
UART / SC16IS
    -> board transport
        -> ModbusRtuMaster
            -> FieldSensor service
                -> диагностический status
                -> Modbus TCP holding map
```

Пример прохождения данных X2X:

```text
UART0
    -> ModbusRtuMaster
        -> X2X service
            -> driver конкретного типа
                -> runtime-структура модуля
                    -> служебная консоль
```

Пример прохождения платформенной конфигурации:

```text
runtime defaults / verified SD set / USB bundle
    -> normalized LcpConfigBundle
        -> A/B storage service
            -> configuration generation
                -> FieldSensor / Ethernet / X2X reconfiguration
```

### 4.3 Владение интерфейсами

Один физический интерфейс в каждый момент времени имеет одного владельца.

```text
UART0       -> X2X master
                либо fallback echo при пустой X2X-конфигурации

S1..S4      -> FieldSensor Modbus RTU master

HMI         -> неблокирующий диагностический echo-test

ETH1        -> отдельный Modbus TCP server
ETH2        -> отдельный Modbus TCP server

USB CDC     -> бинарный configuration transport
                либо текстовая service console
```

> ⚠️ **Внимание**
>
> На одном half-duplex UART нельзя одновременно запускать два protocol engine. Перед подключением нового сервиса необходимо определить владельца порта и отключить прежний режим работы. Для передачи HMI прикладному протоколу его необходимо исключить из диагностического echo-test.

Бинарный USB-режим и текстовая консоль используют один CDC-порт, но не работают с одним входным потоком одновременно. Бинарный кадр с сигнатурой `00 4C 43 50` переводит транспорт в machine mode до команды `EXIT`, закрытия CDC или reset.

### 4.4 Правила разделения

Низкоуровневый драйвер не должен зависеть от прикладной структуры данных. В частности:

- W5500 HAL не должен знать карту FieldSensor;
- Modbus RTU master не должен знать модель прибора;
- X2X-драйвер не должен самостоятельно читать microSD или Flash;
- конфигурационные transport и storage не должны напрямую изменять runtime сервисов;
- служебная консоль не должна изменять protocol state machine в обход публичного API;
- прикладной сервис не должен напрямую управлять регистрами MCU, если для этого существует HAL или board layer;
- пользовательские рецепты и уставки не должны без необходимости расширять платформенную schema-v1.

Такое разделение упрощает проверку, повторное использование кода и добавление новых устройств.

## 5. Модель исполнения FreeRTOS

### 5.1 Основной цикл

Прошивка использует одну основную прикладную задачу LCP:

```text
main.cpp
  -> app_rtos_start()
      -> LCP FreeRTOS task
          -> setup()
          -> постоянный цикл
              -> loop()
              -> vTaskDelay(1 ms)
```

Внутри `loop()` подсистемы обслуживаются в определённом порядке:

```text
PLC OK LED
USB machine configuration transport
USB text console, если machine mode не активен
microSD service
A/B configuration service
инициализация runtime services после выбора конфигурации
X2X service
FieldSensor S1–S4
Ethernet ETH1/ETH2
X2X fallback echo
HMI diagnostic echo
battery
RTC
watchdog
```

USB machine transport проверяется раньше текстовой консоли. После открытия CDC действует короткий guard 250 ms, чтобы первый бинарный кадр не был интерпретирован как текстовая команда.

При наличии валидного внутреннего Flash-слота FieldSensor, Ethernet и X2X запускаются немедленно. Если постоянной конфигурации нет, прошивка ожидает до 2 секунд возможного первоначального SD-импорта, после чего запускает сервисы на runtime defaults.

### 5.2 Неблокирующая модель

Длительная операция представляется машиной состояний:

```text
IDLE
    -> проверить возможность запуска
    -> начать операцию
    -> перейти в WAIT
    -> вернуть управление

WAIT
    -> проверить состояние
    -> при необходимости продолжить ожидание
    -> обработать результат либо timeout
    -> перейти в IDLE
```

Каждый сервис должен иметь:

- явное runtime-состояние;
- функцию инициализации;
- неблокирующую функцию `poll()`;
- ограничение объёма работы за один вызов;
- timeout для аппаратных и протокольных операций;
- явный код результата;
- определённое поведение при ошибке и восстановлении.

A/B configuration service программирует не более одной Flash-страницы 256 байт за один `poll()`. USB transport обрабатывает ограниченные RX/TX chunks. Modbus RTU и TCP сохраняют состояние между вызовами.

> ⚠️ **Внимание**
>
> Блокирующая функция внутри основной задачи останавливает обслуживание X2X, FieldSensor, Ethernet, RTC, HMI, служебной консоли и watchdog на всё время ожидания. Не допускаются длительные `delay`, busy-wait и циклы ожидания аппаратного события без timeout.

### 5.3 Почему используется одна прикладная задача

Единая задача обеспечивает:

- однозначное владение UART и SPI;
- отсутствие конкурирующего доступа к протокольным объектам;
- предсказуемый порядок вызовов;
- минимальное количество mutex и очередей;
- простую диагностику времени выполнения;
- контролируемое использование памяти.

Параметры задачи LCP:

```text
priority = 2
stack = 2048 words = 8192 bytes
delay after each loop = 1 ms
```

Отдельную FreeRTOS-задачу следует вводить только при наличии обоснованной необходимости: независимого временного ограничения, блокирующего внешнего API либо значительного объёма вычислений, который невозможно корректно разделить на шаги.

### 5.4 Контроль памяти

Команда:

```text
rtos
```

показывает:

- общий объём SRAM;
- назначенные и свободные linker-секции;
- `.data` и `.bss`;
- startup stack и зарезервированный C heap;
- размер FreeRTOS heap;
- текущий и минимальный свободный heap;
- размер stack основной задачи;
- пиковое использование stack;
- значения в байтах, words и процентах.

Признаки нормальной работы:

- текущий свободный heap не уменьшается после одинаковых операций;
- minimum-ever-free стабилизируется после начального прогона;
- stack сохраняет достаточный резерв;
- uptime не сбрасывается без штатной причины.

## 6. Сборка и прошивка

### 6.1 Проект

Проект Microchip Studio:

```text
00_LCP2116/RTOS/LCP_Basic.cppproj
```

Целевая платформа:

```text
ATSAM3X8E
ARM Cortex-M3
```

Проект формирует:

```text
LCP_Basic.elf
LCP_Basic.hex
LCP_Basic.bin
LCP_Basic.srec
LCP_Basic.lss
```

### 6.2 Рекомендуемый порядок сборки

1. Выполнить `Clean Solution`.
2. Собрать `Debug`.
3. Проверить ошибки и предупреждения.
4. Выполнить `Clean Solution`.
5. Собрать `Release`.
6. Проверить использование Flash и SRAM.
7. Прошить Release-бинарник.
8. Выполнить минимальный smoke test.
9. Проверить USB configuration workflow.

В Build Output должны присутствовать:

```text
Build succeeded
0 failed
```

Дополнительно фиксируются:

```text
compiler warnings
text / Flash
.data
.bss / SRAM
```

> 🟦 **Информация**
>
> Стандартное сообщение Microchip Studio `Memory Usage estimation may not be accurate...` относится к способу оценки секций ELF и не является предупреждением исходного кода.

### 6.3 Зарезервированная область конфигурации

Linker script исключает последние 32 KiB Flash из области программы:

```text
application ROM = 0x00080000 ... 0x000F7FFF
configuration   = 0x000F8000 ... 0x000FFFFF
RAM             = 0x20000000 ... 0x20017FFF
```

Linker `ASSERT` проверяет полный Flash load image, включая initial image секции `.relocate`, и останавливает сборку при пересечении с конфигурационной областью.

> ⚠️ **Внимание**
>
> Нельзя возвращать всю Flash в application region без одновременного удаления или переноса A/B-хранилища. Иначе новая программа может перезаписать конфигурационные слоты.

### 6.4 Добавление нового исходного файла

После создания нового `.cpp` его необходимо включить в `LCP_Basic.cppproj` через Microchip Studio либо вручную:

```xml

```

Заголовочные файлы подключаются через `#include`, однако их рекомендуется также отображать в дереве проекта для удобства навигации.

> ⚠️ **Внимание**
>
> Наличие файла в каталоге репозитория не означает, что он включён в сборку Microchip Studio. После добавления нового `.cpp` необходимо проверить фактическую команду компилятора в Build Output.

## 7. Платформенная конфигурация

### 7.1 Назначение и границы schema-v1

Платформенная конфигурация содержит только параметры, необходимые базовым аппаратным сервисам LCP2116:

```text
4 x baudrate S1–S4
4 x parity S1–S4
ETH1 MAC, IP, subnet, gateway
ETH2 MAC, IP, subnet, gateway
список до 6 X2X-модулей
4 совместимых значения EIA
32 зарезервированных байта
```

Нормализованный бинарный `LcpConfigBundle` имеет фиксированный размер 104 байта и выравнивание 4 байта.

Поля `EIA` сохраняются для совместимости с ранее использовавшимся комплектом. В текущем baseline FieldSensor логические потоки закреплены непосредственно за S1–S4, поэтому `EIA` не выполняет маршрутизацию портов.

Платформенная schema-v1 не определяет:

- пользовательские рецепты;
- алгоритмы управления;
- уставки технологического процесса;
- карты сигналов конкретного объекта;
- PNX-проект;
- внутреннюю модель будущей Studio.

Разработчик организует прикладную конфигурацию независимо: в исходном коде, собственном бинарном формате, FRAM, microSD, через Ethernet или другим способом.

### 7.2 Бинарная структура `LcpConfigBundle`

Все многобайтовые поля передаются little-endian.

| Смещение | Размер | Поле |
|---:|---:|---|
| 0 | 16 | `field_baudrate[4]`, четыре `uint32_t` |
| 16 | 4 | `field_parity[4]`, `0=8N1`, `1=8O1`, `2=8E1` |
| 20 | 12 | `ethernet_mac[2][6]` |
| 32 | 8 | `ethernet_ip[2][4]` |
| 40 | 8 | `ethernet_subnet[2][4]` |
| 48 | 8 | `ethernet_gateway[2][4]` |
| 56 | 1 | `x2x_module_count` |
| 57 | 6 | `x2x_module_id[6]`, неиспользуемые элементы равны нулю |
| 63 | 1 | `eia_count`, в schema-v1 равно 4 |
| 64 | 8 | `eia_value[4]`, четыре `int16_t` |
| 72 | 32 | `reserved[32]`, должны быть равны нулю |

Основные проверки:

- baudrate S1–S4: `300..1000000`;
- parity: `0`, `1` или `2`;
- MAC: не нулевой и не multicast;
- IP: не `0.0.0.0`, не multicast и не `255.255.255.255`;
- subnet: ненулевая непрерывная маска;
- X2X count: не более 6;
- каждый активный X2X ID существует в каталоге и разрешён для конфигурации;
- `eia_count = 4`, каждое значение `0..255`;
- reserved bytes равны нулю.

### 7.3 Приоритет источников при запуске

```text
1. Наиболее новый валидный слот внутренней Flash.
2. Автоматический импорт с microSD, только если оба Flash-слота невалидны.
3. Встроенные runtime defaults, если постоянная конфигурация отсутствует.
```

При наличии валидного Flash-слота runtime-сервисы запускаются сразу.

При отсутствии Flash прошивка ожидает до 2 секунд возможного первоначального SD-импорта. Если стабильный полный комплект не получен, FieldSensor, Ethernet и X2X запускаются на defaults.

Если карты при старте не было, автоматический import может выполниться после её более поздней установки, пока initial import остаётся pending. Если карта была доступна, но установленный комплект оказался неполным или невалидным, после исправления файлов следует использовать явную команду `config import sd`.

Существующая Flash-конфигурация никогда не заменяется автоматически отличающимся содержимым microSD. Для обновления требуется явный `config import sd` либо запись через бинарный USB-протокол.

Каждое фактическое изменение активного bundle увеличивает `generation`. FieldSensor, Ethernet и X2X отслеживают generation и безопасно применяют новые параметры.

### 7.4 Встроенные runtime defaults

```text
S1 = 9600 8N1
S2 = 9600 8N1
S3 = 9600 8N1
S4 = 9600 8N1

ETH1 MAC = 02:4C:43:50:00:01
ETH1 IP = 192.168.1.1
ETH1 subnet = 255.255.255.0
ETH1 gateway = 192.168.1.254

ETH2 MAC = 02:4C:43:50:00:02
ETH2 IP = 192.168.1.2
ETH2 subnet = 255.255.255.0
ETH2 gateway = 192.168.1.254

X2X modules = none
EIA = 3, 4, 5, 6
```

Defaults применяются только в runtime и автоматически во Flash не записываются. Нормальное состояние нового контроллера:

```text
source = defaults
slot = none
sequence = 0
```

Команда:

```text
config defaults
```

только выводит встроенный профиль. Она не изменяет активный bundle и не программирует Flash.

> ⚠️ **Внимание**
>
> Одинаковые default IP и MAC предназначены для одиночного контроллера на изолированном стенде. Перед подключением нескольких новых контроллеров к общей Ethernet-сети каждому устройству необходимо назначить уникальные адреса.

### 7.5 Внутренняя Flash и A/B-слоты

Последние 32 KiB внутренней Flash:

```text
0x000F8000 ... 0x000FFFFF
```

разделены на два слота:

```text
Slot A = 16 KiB
Slot B = 16 KiB
```

Одна запись включает:

1. 32-байтовый record header;
2. 104-байтовый `LcpConfigBundle`;
3. CRC32 payload;
4. отдельную 32-байтовую commit record в последней Flash-странице слота;
5. read-back verification.

Новая конфигурация всегда записывается в неактивный слот. Commit page записывается последней. При старте прошивка проверяет header, schema, размер, CRC, semantic validation и commit record обоих слотов, затем выбирает валидную запись с наиболее новым wrap-safe `sequence`.

Если питание исчезло до корректной commit page, незавершённый слот игнорируется, а предыдущее валидное поколение остаётся доступным.

Повторная передача полностью совпадающего bundle возвращает:

```text
no change
```

При этом Flash не программируется, slot и sequence не изменяются.

Команды:

```text
config status
config rollback
config erase confirm
```

`config rollback` публикует предыдущее валидное содержимое как новое поколение. `config erase confirm` неблокирующе инвалидирует оба слота и после завершения erase активирует runtime defaults. Автоматический SD import в этой же runtime-сессии не запускается повторно; после следующего reboot пустая Flash снова рассматривается как новый контроллер и initial auto-import разрешается.

### 7.6 Опциональный импорт с microSD

microSD поддерживает FAT16/FAT32. Пример полного комплекта находится в:

```text
00_LCP2116/RTOS/sd_card/
```

В корне карты должны одновременно находиться все 12 файлов:

```text
baud.TXT
Parity.TXT
MAC.txt
IP.txt
SUBNET.txt
GATE.txt
MAC2.txt
IP2.txt
SUBNET2.txt
GATE2.txt
X2X.TXT
EIA.TXT
```

Комплект принимается только целиком. Ошибка одного обязательного файла отклоняет весь candidate и не изменяет активную конфигурацию.

Общий числовой формат:

```text
COUNT
VALUE_1
VALUE_2
...
fin
```

Для файлов `baud`, `Parity`, сетевых параметров и `EIA` общий numeric parser поддерживает:

- пустые строки;
- пробелы и tab в начале и конце;
- комментарии после `//`;
- комментарии после `#`;
- UTF-8 BOM в начале;
- `fin` в любом регистре;
- строгий десятичный ввод со знаком, когда тип его допускает.

`X2X.TXT` обрабатывается отдельным parser. Он поддерживает пустые строки, пробелы, комментарии `//` и `Fin` без учёта регистра. Для совместимости не следует добавлять в `X2X.TXT` UTF-8 BOM или комментарии `#`.

Первая непустая строка является количеством значений. Количество и фактическое число значений до `fin` должны совпадать.

Имена файлов являются частью внешнего формата и чувствительны к регистру:

```text
правильно:   baud.TXT, Parity.TXT, IP.txt
неправильно: BAUD.TXT, PARITY.TXT, IP.TXT
```

Перед Flash-записью сервис читает весь комплект дважды. Оба нормализованных bundle и их CRC должны полностью совпасть.

Команды:

```text
sd
sd test
config scan sd
config compare sd
config import sd
config show [FILE|all]
```

Назначение:

- `sd` — наличие карты, готовность filesystem и тип FAT;
- `sd test` — запись и чтение `SDTEST.TXT`;
- `config scan sd` — два чтения и полная проверка без записи;
- `config compare sd` — проверка и список отличающихся файлов;
- `config import sd` — проверка и атомарная запись в неактивный слот;
- `config show all` — вывод активного bundle в каноническом TXT-виде.

> ⚠️ **Внимание**
>
> `sd test` изменяет содержимое карты. Команда предназначена для сервисной проверки и создаёт тестовый файл.

### 7.7 Бинарный USB configuration protocol

Машинный протокол использует тот же USB CDC-порт, что и текстовая консоль. Кадр начинается с:

```text
00 4C 43 50
```

Основные параметры:

```text
protocol version = 1
configuration schema = 1
storage format = 1
header size = 24 bytes
maximum payload = 160 bytes
maximum frame = 184 bytes
byte order = little-endian
```

#### Заголовок кадра

| Смещение | Размер | Поле |
|---:|---:|---|
| 0 | 4 | magic `00 4C 43 50` |
| 4 | 1 | protocol version |
| 5 | 1 | command |
| 6 | 1 | status; в request должен быть 0 |
| 7 | 1 | flags |
| 8 | 4 | request ID |
| 12 | 4 | payload length |
| 16 | 4 | payload CRC32 |
| 20 | 4 | header CRC32 по байтам `0..19` |

Ответ повторяет `command` и `request ID` запроса, помещает результат в поле `status` и использует нулевые flags. Request с ненулевым полем `status` отклоняется. Поле `flags` в request зарезервировано; host protocol version 1 должен передавать 0.

CRC32 соответствует стандартному алгоритму, используемому `zlib.crc32` и библиотекой `lcp_crc32`.

#### Команды

```text
1 HELLO
2 GET_CONFIG
3 VALIDATE_CONFIG
4 PUT_CONFIG
5 GET_STATUS
6 REBOOT
7 EXIT
```

#### Статусы

```text
0  OK
1  ACCEPTED
2  BUSY
3  NO_CHANGE
4  INVALID_CRC
5  UNSUPPORTED_VERSION
6  UNSUPPORTED_COMMAND
7  INVALID_LENGTH
8  INVALID_SCHEMA
9  INVALID_CONFIG
10 FLASH_ERROR
11 VERIFY_ERROR
12 REBOOT_REQUIRED
13 INTERNAL_ERROR
```

#### Payload `VALIDATE_CONFIG` и `PUT_CONFIG`

```text
offset 0  uint16 schema_version
offset 2  uint16 bundle_size
offset 4  uint32 bundle_crc32
offset 8  104 bytes LcpConfigBundle
```

Полная длина payload равна 112 байтам.

`VALIDATE_CONFIG` выполняет проверку без записи. `PUT_CONFIG` сначала проверяет schema, размер, CRC и значения, затем:

- возвращает `NO_CHANGE`, если bundle полностью совпадает с активным;
- либо немедленно возвращает `ACCEPTED` и запускает неблокирующую запись;
- host опрашивает `GET_STATUS`; сам кадр ответа имеет status `OK`, а `writer state` и `writer result` находятся в payload;
- после записи record page, commit page и read-back verification `writer state` становится `COMPLETE`, а `writer result` — `REBOOT_REQUIRED`. До reboot поля active source, active slot, active sequence, active CRC32 и generation относятся к текущему runtime; новая запись описывается target slot и target sequence.

#### Ответ `HELLO`

Payload 16 байт:

| Смещение | Размер | Поле |
|---:|---:|---|
| 0 | 2 | protocol version |
| 2 | 2 | schema version |
| 4 | 2 | bundle size |
| 6 | 2 | maximum payload |
| 8 | 4 | capabilities; `0x0000000F` |
| 12 | 2 | storage version |
| 14 | 2 | header size |

Capabilities в версии 1:

```text
bit 0 read
bit 1 validate
bit 2 write
bit 3 reboot
```

#### Ответ `GET_CONFIG`

Payload состоит из 20 байт metadata и 104 байт bundle:

| Смещение | Размер | Поле |
|---:|---:|---|
| 0 | 2 | schema version |
| 2 | 2 | bundle size |
| 4 | 4 | active sequence |
| 8 | 4 | generation |
| 12 | 4 | bundle CRC32 |
| 16 | 1 | source: `0=defaults`, `1=internal Flash` |
| 17 | 1 | slot: `0=none`, `1=A`, `2=B` |
| 18 | 2 | reserved |
| 20 | 104 | `LcpConfigBundle` |

#### Ответ `GET_STATUS`

Payload 28 байт:

| Смещение | Размер | Поле |
|---:|---:|---|
| 0 | 1 | writer state |
| 1 | 1 | writer result |
| 2 | 1 | active source |
| 3 | 1 | active slot |
| 4 | 1 | target slot |
| 5 | 1 | configuration service busy |
| 6 | 2 | reserved |
| 8 | 4 | active sequence |
| 12 | 4 | target sequence |
| 16 | 4 | active CRC32 |
| 20 | 4 | generation |
| 24 | 1 | Flash result |
| 25 | 3 | reserved |

Writer states:

```text
0 IDLE
1 WRITE_RECORD
2 WRITE_COMMIT
3 VERIFY
4 COMPLETE
5 ERROR
```

`REBOOT` сначала передаёт ответ `OK`, затем через 100 ms выполняет `NVIC_SystemReset()`. `EXIT` завершает machine session после передачи ответа и возвращает порт текстовой консоли.

> ⚠️ **Внимание**
>
> Host не должен вслепую повторять `PUT_CONFIG` после неопределённого обрыва связи: запрос мог быть принят и выполнен. После такой ситуации необходимо повторно подключиться, выполнить `GET_CONFIG` и сравнить активный bundle.

### 7.8 Reference Python utility

Пользовательская точка входа:

```text
tools/lcp_config_usb.py
```

Внутренний модуль реализации протокола и schema:

```text
tools/_lcp_config_usb_core.py
```

Пользователь не должен запускать внутренний модуль напрямую.

Требования:

```text
Python 3.9+
pyserial
```

Команды:

```text
info        информация о protocol/schema и writer status
read        чтение активной конфигурации
write       запись JSON-модели
write-txt   импорт полного TXT-каталога и запись
import-txt  преобразование TXT в JSON без контроллера
export-txt  экспорт JSON в канонический TXT-комплект
```

JSON является только удобным interchange-форматом reference utility. Он не является обязательным форматом прикладного проекта.

Windows CDC может повторно создать COM-порт не мгновенно. Пользовательская утилита выполняет ограниченные повторы открытия и `HELLO`, а после изменённой записи:

1. отправляет одну команду `REBOOT`;
2. ждёт исчезновения старого COM;
3. находит контроллер после повторного появления;
4. выполняет `HELLO`;
5. читает активный bundle;
6. сравнивает его с отправленными данными.

Успешный результат:

```text
reboot required
reconnected and verified: source=1 slot=... sequence=... crc32=0x........
```

Повторная запись совпадающих данных:

```text
no change
```

Опция `--no-reboot` записывает новый неактивный слот, но оставляет текущий runtime без изменений до следующего reset или отключения питания.

## 8. Служебная консоль USB

### 8.1 Подключение

Прошивка создаёт USB CDC-порт. Для подключения можно использовать PuTTY, Tera Term, RealTerm или другой последовательный терминал.

Рекомендуемые настройки:

```text
115200
8 data bits
no parity
1 stop bit
no flow control
```

Для USB CDC выбранный baudrate не определяет физическую скорость передачи, однако 115200 используется как стандартная настройка терминала.

После подключения отображается:

```text
LCP firmware ready. Type 'help' or '?'.
```

Команды не чувствительны к регистру и выполняются клавишей Enter.

### 8.2 Формат вывода

Именованные параметры:

```text
name = value
name = value, next = value
```

Индексированные массивы:

```text
0: value, 1: value
```

Двоеточия внутри значения сохраняются:

```text
mac = DE:AD:BE:EF:FE:EE
datetime = 2026-07-20 21:30:00
```

Общие функции форматирования находятся в:

```text
app/diagnostics/diagnostic_output.hpp
```

### 8.3 Работа со справкой

Основная команда:

```text
help
```

Короткая форма:

```text
?
```

`help` выводит сгруппированный список доступных команд.

Для получения полного состояния используется:

```text
status
```

Для регулярной диагностики рекомендуется вызывать профильные команды `field`, `x2x`, `eth`, `rtc`, `rtos` и другие. Это сокращает объём вывода и облегчает анализ конкретной подсистемы.

> 🟦 **Информация**
>
> При копировании большого отчёта из терминала необходимо учитывать настройки переноса строк. Для корректного сохранения журнала рекомендуется использовать terminal logging либо окно шириной не менее 80 символов.

### 8.4 Совместная работа с машинным USB-протоколом

Текстовая консоль и бинарный configuration transport используют один USB CDC-порт.

После открытия CDC текстовая консоль удерживается 250 ms. Это даёт host-приложению время передать первый бинарный кадр с сигнатурой `00 4C 43 50`. Пока machine session активна, принятые байты не передаются текстовому parser.

Текстовый режим восстанавливается после:

- команды `EXIT`;
- закрытия USB CDC;
- reset контроллера.

Терминал и Python-утилита не должны одновременно удерживать один COM-порт.

### 9.9 Универсальный интерфейс HMI

HMI является отдельным пользовательским интерфейсом RS-485. Название разъёма отражает первоначальное назначение, но программно порт не закреплён за панелью оператора.

В базовой прошивке HMI используется для неблокирующего диагностического echo-test со следующими параметрами:

```text
baudrate = 9600
frame = 8N1
interframe gap = 5 ms
response delay = 10 ms
maximum frame size = 128 bytes
maximum RX processing per poll = 16 bytes
TX timeout = 1000 ms
```

Алгоритм работы:

1. В каждом цикле `poll()` из FIFO принимается ограниченное количество байтов.
2. После паузы не менее 5 ms принятый набор байтов считается завершённым кадром.
3. Через 10 ms начинается возврат кадра.
4. Если FIFO передатчика принимает только часть данных, передача продолжается в следующих циклах с сохранённого смещения.
5. Если передача не может быть завершена за 1000 ms, кадр сбрасывается без блокировки основной задачи.

Поддерживаются две аппаратные схемы внешних UART:

```text
двухканальное устройство на UART2_CS
    HMI -> channel A
    S1  -> channel B

два одноканальных устройства
    HMI -> UART2_CS, channel A
    S1  -> UART3_CS, channel A
```

Выбор выполняется автоматически при старте по результатам проверки доступных каналов. Прикладной код HMI использует логический descriptor и не зависит от установленного варианта SC16IS.

Основные файлы реализации:

```text
board/lcp_sc16is.hpp
board/lcp_sc16is.cpp
hal/sc16is7xx.hpp
hal/sc16is7xx.cpp
app/diagnostics/sc16is_echo_test.hpp
app/diagnostics/sc16is_echo_test.cpp
```

Назначение файлов:

```text
lcp_sc16is.*
    определение аппаратной конфигурации и построение логической карты HMI/S1

sc16is7xx.*
    SPI-доступ, настройка baudrate/frame, FIFO и автоматического RS-485 RTS

sc16is_echo_test.*
    неблокирующий приём кадра и его возврат на HMI
```

> 🟦 **Информация**
>
> В версии 1.02.0 HMI не входит в FieldSensor S1–S4 и не публикуется в карте Modbus TCP `0..11`. Echo-test подтверждает работу физического интерфейса, приёмника, передатчика, SPI-связи с SC16IS и автоматического управления направлением RS-485.

Для использования HMI прикладным протоколом необходимо:

1. исключить HMI из инициализации `sc16is_echo_test_init()`;
2. получить логический дескриптор `lcp_sc16is_get_map().hmi`;
3. настроить порт через `sc16is_begin()`;
4. создать отдельный неблокирующий service с функциями `init()` и `poll()`;
5. реализовать timeout, обработку частичного TX и очистку RX;
6. добавить профильную диагностику;
7. проверить, что echo-test и новый service не обслуживают HMI одновременно.

> ⚠️ **Внимание**
>
> Передача HMI новому сервису должна выполняться как смена владельца интерфейса. Одновременная работа echo-test и прикладного протокола на одном канале SC16IS не допускается.

## 10. FieldSensor и порты S1–S4

FieldSensor представляет собой базовый пример подключения внешнего Modbus RTU-прибора к пользовательским портам контроллера.

Все четыре порта работают как независимые Modbus RTU master и по умолчанию выполняют одинаковый запрос:

```text
slave address = 1
function = FC03 Read Holding Registers
start register = 0
register count = 2
poll period = 300 ms
timeout = 500 ms
interframe gap = 5 ms
connection loss threshold = 5 consecutive failures
```

Команды:

```text
field
field reload
field pause
field resume
```

### 10.1 Физическое соответствие

```text
S1 -> SC16IS7xx
S2 -> ATSAM3X8E UART1 / Serial1
S3 -> ATSAM3X8E UART3 / Serial3
S4 -> ATSAM3X8E UART2 / Serial2
```

Для S1 подходящий канал SC16IS определяется автоматически.

### 10.2 baud.TXT

```text
4
9600
9600
1200
9600
fin
```

Значения расположены в порядке:

```text
S1
S2
S3
S4
```

В файле указываются фактические baudrate, а не индексы таблицы.

### 10.3 Parity.TXT

```text
4
0
0
0
0
fin
```

Коды формата:

```text
0 = 8N1
1 = 8O1
2 = 8E1
```

В версии 1.02.0 эти файлы являются частью полного импортируемого bundle. При обычной работе FieldSensor получает baudrate и parity из активной внутренней конфигурации. Ошибка любого обязательного файла отклоняет весь новый SD candidate и не изменяет текущие параметры.

### 10.4 Runtime каждого порта

Каждый S1–S4 имеет собственные:

- transport callbacks;
- объект `ModbusRtuMaster`;
- RX/TX buffers протокола;
- два регистра результата;
- таймер следующего запроса;
- timeout;
- признаки качества;
- счётчики успешных и неуспешных транзакций;
- время последнего успешного обновления.

Отдельные FreeRTOS-задачи для портов не создаются. Один `field_sensor_service_poll()` последовательно выполняет неблокирующий шаг для каждого порта.

### 10.5 Диагностика

Команда:

```text
field
```

показывает для каждого порта:

- роль;
- наличие transport;
- фактический baudrate и формат;
- slave address;
- start address и register count;
- period и timeout;
- состояние connection/valid/request;
- последние принятые регистры;
- success/failed counters;
- consecutive failures;
- UART errors;
- время последнего успешного обновления.

Нормальный подключённый прибор:

```text
connection = online
valid = yes
last_result = ok
consecutive_failures = 0
uart_errors = 0
```

После пяти последовательных ошибок:

```text
connection = lost
valid = no
```

> 🟦 **Информация**
>
> При потере связи последние корректно принятые значения сохраняются. Они представляют собой last-good data и могут использоваться только совместно с признаком `valid = yes`.

### 10.6 Reload и pause

```text
field reload
```

Команда запрашивает повторное применение serial-параметров из текущего активного `LcpConfigBundle`. Она не читает `baud.TXT` и `Parity.TXT` напрямую. Активные транзакции сначала завершаются, после чего UART переинициализируются. Переинициализация сбрасывает runtime values, quality и counters каждого порта.

```text
field pause
field resume
```

Pause запрещает запуск новых запросов, но не обрывает активный RTU-кадр.

### 10.7 Проверка порта

1. Подключить Modbus RTU slave.
2. Установить slave address 1.
3. Настроить baudrate и parity через активный bundle: USB provisioning либо полный SD import.
4. Выполнить `field`.
5. Проверить переход в `online` и рост `success`.
6. Изменить значения регистров и проверить их обновление.
7. Отключить линию.
8. Проверить `connection = lost` и `valid = no`.
9. Восстановить линию.
10. Проверить автоматический возврат в `online`.

## 11. Подключение другого Modbus RTU-прибора

### 11.1 Выбор способа интеграции

Для простого прибора, который опрашивается одним запросом FC03, можно изменить существующий FieldSensor example.

Для устройства с несколькими запросами, операциями записи, сложным декодированием или собственной логикой рекомендуется создать отдельный прикладной service.

Универсальный `ModbusRtuMaster` при этом не изменяется.

> 🟦 **Информация**
>
> Платформенная schema-v1 изменяет только физические baudrate и parity S1–S4. Slave address, function code, register map, период, timeout и прикладное декодирование остаются частью пользовательского service.

### 11.2 Параметры текущего примера

Файл:

```text
app/field/field_sensor_service.cpp
```

Функция:

```cpp
set_default_configs()
```

Основные параметры:

```cpp
g_configs[index].slave_address
g_configs[index].start_register
g_configs[index].register_count
g_configs[index].poll_period_ms
g_configs[index].timeout_ms
```

Пример:

```cpp
g_configs[index].role = FIELD_PORT_MASTER;
g_configs[index].slave_address = 7U;
g_configs[index].start_register = 100U;
g_configs[index].register_count = 4U;
g_configs[index].poll_period_ms = 1000U;
g_configs[index].timeout_ms = 700U;
```

### 11.3 Размер буфера

Текущее значение:

```cpp
FIELD_SENSOR_REGISTER_COUNT = 2U
```

Файл:

```text
app/field/field_sensor_service.hpp
```

Если прибор возвращает больше регистров, необходимо одновременно проверить:

1. `FIELD_SENSOR_REGISTER_COUNT`;
2. `FieldSensorPortState::registers`;
3. `set_default_configs()`;
4. декодирование данных;
5. `field_status.cpp`;
6. `ethernet_modbus_service.*`;
7. `ethernet_status.cpp`;
8. Modbus TCP-документацию.

> ⚠️ **Внимание**
>
> Нельзя увеличить только `register_count`, оставив прежний runtime buffer. Такое изменение создаёт риск записи за границы массива.

### 11.4 Универсальные ограничения baseline master

```text
slave address = 1..247
FC03 maximum = 100 registers
FC10 maximum = 16 registers
RX frame buffer = 256 bytes
TX frame buffer = 64 bytes
one active transaction per ModbusRtuMaster
```

`ModbusRtuMaster` не владеет физическим transport и внешним FC03 output buffer. Transport и output array должны существовать до завершения транзакции. Для каждого физического UART создаётся отдельный master object.

### 11.5 Когда необходим отдельный service

Отдельный service рекомендуется, если прибор требует:

- нескольких последовательных запросов;
- FC10 или других операций записи;
- собственной state machine;
- преобразования нескольких типов данных;
- масштабирования и проверки диапазона;
- отдельных quality-флагов;
- собственной диагностики;
- независимой карты публикации.

### 11.6 Неблокирующий шаблон Modbus RTU service

```text
1. вызвать modbus_rtu_master_poll()
2. проверить завершение активной транзакции
3. обработать result и exception code
4. сохранить новые данные только после полной проверки ответа
5. запланировать следующий запрос
6. при наступлении срока запустить новую транзакцию
7. вернуть управление основной задаче
```

Не допускаются:

```text
delay(...)
while (ожидание ответа)
busy-wait аппаратного флага
ожидание без timeout
```

### 11.7 Альтернативный slave-режим

Повторно используемая реализация FC03 slave находится в:

```text
protocol/modbus_rtu/modbus_rtu_slave.cpp
protocol/modbus_rtu/modbus_rtu_slave.hpp
```

Отключённый пример подключения:

```text
app/field/field_sensor_slave_example.cpp
```

При включении slave-режима необходимо изменить владельца соответствующего физического порта и отключить master на этом же UART.

## 12. Ethernet и Modbus TCP

Контроллер имеет два независимых W5500. Каждый интерфейс обслуживается отдельным экземпляром `ModbusTcpServer`, собственными RX/TX buffers и счётчиками.

```text
protocol = Modbus TCP
port = 502
function = FC03 Read Holding Registers
holding registers = 0..11
write functions = not supported
```

ETH1 и ETH2 публикуют одинаковую карту данных FieldSensor.

### 12.1 Конфигурационные файлы

ETH1:

```text
MAC.txt
IP.txt
SUBNET.txt
GATE.txt
```

ETH2:

```text
MAC2.txt
IP2.txt
SUBNET2.txt
GATE2.txt
```

Формат IP, SUBNET и GATE:

```text
4
192
168
1
1
fin
```

Формат MAC:

```text
6
100
200
249
154
182
220
fin
```

MAC задаётся шестью десятичными байтами.

> ⚠️ **Внимание**
>
> Одинаковый IP на ETH1 и ETH2 допустим только при подключении интерфейсов к физически раздельным сетям. При работе в одной LAN должны использоваться разные IP-адреса и уникальные MAC-адреса.

### 12.2 Применение настроек

```text
eth reload
```

Команда повторно применяет сетевые параметры из текущего активного `LcpConfigBundle` и переинициализирует оба W5500. Она не читает восемь файлов напрямую. Аппаратный reset закрывает активные TCP-соединения; клиенты должны подключиться повторно.

Проверка:

```text
eth
```

Ожидаемое аппаратное состояние подключённого интерфейса:

```text
initialized = yes
init = ok
VERSIONR = 0x04
link = up
```

### 12.3 Holding register map

| Адрес | Значение |
|---:|---|
| 0 | S1 value0 |
| 1 | S1 value1 |
| 2 | S1 quality |
| 3 | S2 value0 |
| 4 | S2 value1 |
| 5 | S2 quality |
| 6 | S3 value0 |
| 7 | S3 value1 |
| 8 | S3 quality |
| 9 | S4 value0 |
| 10 | S4 value1 |
| 11 | S4 quality |

Чтение всей карты:

```text
FC03
start address = 0
quantity = 12
```

> 🟦 **Информация**
>
> Некоторые Modbus-клиенты отображают holding registers как `40001`, `40002` и далее. В таком представлении `40001` обычно соответствует protocol offset `0`.

### 12.4 Quality register

```text
bit 0      valid
bit 1      connection_lost
bit 2      physical port present
bit 3      RTU request active
bit 4      FieldSensor service paused
bits 8..15 ModbusRtuResult
```

Значения `ModbusRtuResult`:

```text
0 idle
1 busy
2 ok
3 timeout
4 CRC error
5 invalid response
6 Modbus exception
7 transport error
8 invalid argument
```

> ⚠️ **Внимание**
>
> Клиент обязан анализировать quality register. Значения `value0` и `value1` могут содержать последние корректные данные после потери RS-485 связи.

### 12.5 Диагностика

```text
eth
```

Проверяются:

```text
requests = responses
malformed = 0
transport_errors = 0
```

`exceptions` может увеличиваться при запросе неверного адреса, недопустимого количества регистров или неподдерживаемой функции.

### 12.6 Расширение карты

Изменяются:

```text
app/ethernet/ethernet_modbus_service.hpp
app/ethernet/ethernet_modbus_service.cpp
app/diagnostics/ethernet_status.cpp
README.md
```

Порядок расширения:

1. определить новую карту адресов;
2. увеличить статический holding buffer;
3. заполнить новые адреса в `update_holding_map()`;
4. проверить диапазоны в `read_holding()`;
5. обновить compile-time `static_assert`;
6. обновить диагностический вывод;
7. обновить соответствующий раздел настоящего README;
8. протестировать минимальный и максимальный адрес;
9. проверить exception response за пределами карты.

Общие Modbus TCP function codes реализуются в `protocol/modbus_tcp/`. Прикладная карта и callbacks находятся в `app/ethernet/`. W5500 HAL отвечает только за Ethernet transport.

## 13. Внутренняя шина X2X

X2X используется для последовательного подключения функциональных модулей Lorentz. Контроллер LCP2116 выполняет роль master и циклически обслуживает модули в порядке, заданном активной платформенной конфигурацией. `X2X.TXT` является совместимым входным файлом полного SD-комплекта, но после импорта рабочим источником служит внутренний `LcpConfigBundle`.

Команды:

```text
x2x
x2x reload
x2x pause
x2x resume
x2x ldo SLAVE VALUE
```

### 13.1 Архитектура X2X

```text
active LcpConfigBundle
    -> x2x_config
        -> x2x_registry
            -> x2x_service
                -> driver конкретного типа
                    -> ModbusRtuMaster
```

Назначение компонентов:

```text
x2x_config
    проверка и представление порядка X2X-модулей

x2x_registry
    создание статических экземпляров модулей

x2x_catalog
    описание поддерживаемых типов и callbacks

x2x_service
    последовательный неблокирующий опрос модулей

x2x module driver
    карта регистров и state machine конкретного типа

x2x_status
    общая и специфическая диагностика
```

Baseline timing:

```text
maximum modules = 6
Modbus interframe gap = 5 ms
gap between completed module cycles = 50 ms
one Modbus transaction timeout = 500 ms
connection loss threshold = 5 consecutive failures
```

### 13.2 Формат X2X.TXT и положение в bundle

При SD provisioning первая строка `X2X.TXT` содержит количество модулей. Далее указываются числовые ID в физическом порядке цепочки. После полного успешного импорта эти значения сохраняются в `x2x_module_count` и `x2x_module_id[6]` активного bundle.

```text
4
1
2
5
6
fin
```

Соответствие:

```text
slave 1 -> ID 1
slave 2 -> ID 2
slave 3 -> ID 5
slave 4 -> ID 6
```

Максимальное количество внешних модулей — 6.

Пустая цепочка:

```text
0
fin
```

При пустой цепочке UART0 передаётся fallback echo-test.

### 13.3 Стабильные ID

```text
0 -> LCP2116, запрещён в X2X.TXT
1 -> LDO1118
2 -> LAI1118
3 -> LDI1118
4 -> LCT1114
5 -> LDI1116
6 -> LCT1114_2
```

> ⚠️ **Внимание**
>
> Числовой ID является частью формата `X2X.TXT`. После публикации ID нельзя изменять, удалять или повторно назначать другому типу модуля.

### 13.4 Runtime модуля

Общий `X2XDeviceHeader` содержит:

- type;
- ASDU;
- slave address;
- connection state;
- device fault;
- communication result;
- exception code;
- success/failed counters;
- consecutive failures;
- last update time.

Структура конкретного типа хранит его прикладные данные: дискретные входы, целочисленные измерения, float values, output state либо другие параметры.

### 13.5 Диагностика

```text
x2x
```

Для каждого подключённого модуля проверяются:

```text
connection = online
communication_error = ok
success увеличивается
consecutive_failures = 0
uart_errors = 0
```

После отключения растёт `failed`, затем устанавливается:

```text
connection = lost
```

Первый корректный цикл после восстановления должен вернуть:

```text
connection = online
communication_error = ok
```

### 13.6 Reload и pause

```text
x2x reload
```

Команда повторно применяет X2X-часть текущего активного bundle. Если текущий цикл активен, применение откладывается до безопасного завершения транзакции. Для изменения состава цепочки сначала необходимо записать новый полный bundle через USB либо выполнить `config import sd`.

```text
x2x pause
x2x resume
```

Pause завершает активный цикл модуля и только затем запрещает новые опросы.

### 13.7 Сервисная проверка LDO

```text
x2x ldo 1 255
```

Команда задаёт локальное output value экземпляра LDO1118. Драйвер передаёт значение модулю при следующем цикле опроса.

## 15. Разработка неблокирующего X2X-драйвера

### 15.1 Контекст драйвера

`X2XModuleContext` предоставляет:

```text
master
runtime
waveform
register_buffer
register_capacity
now_ms
```

Драйвер использует общий `ModbusRtuMaster` X2X-сервиса и общий временный register buffer.

> ⚠️ **Внимание**
>
> `x2x_service_poll()` уже вызывает `modbus_rtu_master_poll()` перед callback активного драйвера. Драйвер конкретного модуля не должен повторно вызывать `modbus_rtu_master_poll()`.

### 15.2 Машина состояний

Один вызов драйвера выполняет один шаг:

```text
START_READ
    -> проверить master ready
    -> запустить запрос
    -> перейти в WAIT_READ
    -> вернуть IN_PROGRESS

WAIT_READ
    -> проверить result
    -> если BUSY, вернуть IN_PROGRESS
    -> декодировать данные либо зарегистрировать ошибку
    -> reset master и runtime state
    -> вернуть CYCLE_COMPLETE
```

### 15.3 Рабочий skeleton

```cpp
#include "x2x_module_drivers.hpp"
#include "x2x_module_common.hpp"

namespace
{
enum LtmPollState : uint8_t
{
    LTM_POLL_START_READ = 0U,
    LTM_POLL_WAIT_READ = 1U
};

static const uint16_t LTM_REGISTER_COUNT = 9U;
static const uint16_t LTM_FLOAT_START_REGISTER = 1U;
}

X2XModulePollResult x2x_module_poll_ltm1114(
    X2XDeviceHeader* device,
    X2XModuleContext& context)
{
    if ((device == 0) ||
        (device->type != X2X_DEVICE_LTM1114) ||
        (context.master == 0) ||
        (context.runtime == 0) ||
        (context.register_buffer == 0) ||
        (context.register_capacity < LTM_REGISTER_COUNT))
    {
        return X2X_MODULE_POLL_CYCLE_COMPLETE;
    }

    X2XLtm1114* ltm = static_cast<X2XLtm1114*>(device);

    switch (context.runtime->state)
    {
        case LTM_POLL_START_READ:
            if (modbus_rtu_master_ready(*context.master) == 0U)
            {
                return X2X_MODULE_POLL_IN_PROGRESS;
            }

            if (modbus_rtu_master_start_read_holding(
                    *context.master,
                    device->slave_address,
                    0U,
                    LTM_REGISTER_COUNT,
                    context.register_buffer,
                    X2X_MODBUS_TRANSACTION_TIMEOUT_MS) == 0U)
            {
                x2x_module_mark_failure(
                    *device,
                    modbus_rtu_master_result(*context.master),
                    modbus_rtu_master_exception_code(*context.master));
                modbus_rtu_master_reset(*context.master);
                context.runtime->state = LTM_POLL_START_READ;
                return X2X_MODULE_POLL_CYCLE_COMPLETE;
            }

            context.runtime->state = LTM_POLL_WAIT_READ;
            return X2X_MODULE_POLL_IN_PROGRESS;

        case LTM_POLL_WAIT_READ:
        {
            const ModbusRtuResult result =
                modbus_rtu_master_result(*context.master);

            if (result == MODBUS_RTU_RESULT_BUSY)
            {
                return X2X_MODULE_POLL_IN_PROGRESS;
            }

            if (result == MODBUS_RTU_RESULT_OK)
            {
                ltm->digital_inputs = context.register_buffer[0];
                x2x_module_decode_float_array(
                    ltm->tf_values,
                    X2XLtm1114::TF_COUNT,
                    context.register_buffer,
                    LTM_FLOAT_START_REGISTER,
                    LTM_REGISTER_COUNT);
                x2x_module_mark_success(*device, context.now_ms);
            }
            else
            {
                x2x_module_mark_failure(
                    *device,
                    result,
                    modbus_rtu_master_exception_code(*context.master));
            }

            modbus_rtu_master_reset(*context.master);
            context.runtime->state = LTM_POLL_START_READ;
            return X2X_MODULE_POLL_CYCLE_COMPLETE;
        }

        default:
            modbus_rtu_master_reset(*context.master);
            context.runtime->state = LTM_POLL_START_READ;
            return X2X_MODULE_POLL_CYCLE_COMPLETE;
    }
}
```

Адреса и размер запроса должны соответствовать фактической карте регистров модуля.

### 15.4 Порядок слов float

Штатные функции преобразования используют:

```text
первый регистр  -> младшее 16-битное слово
второй регистр -> старшее 16-битное слово
```

```cpp
x2x_module_registers_to_float(...)
x2x_module_decode_float_array(...)
```

Если новый модуль использует другой порядок слов, преобразование реализуется только в его драйвере.

> ⚠️ **Внимание**
>
> Нельзя глобально изменять общий helper для нового порядка слов: это нарушит декодирование существующих модулей.

### 15.5 Фиксация результата

Успешный обмен:

```cpp
x2x_module_mark_success(*device, context.now_ms);
```

Ошибка:

```cpp
x2x_module_mark_failure(
    *device,
    modbus_rtu_master_result(*context.master),
    modbus_rtu_master_exception_code(*context.master));
```

### 15.6 Проверка нового драйвера

1. Собрать Debug и Release.
2. Создать `X2X.TXT` с новым ID.
3. Записать bundle с новым ID и выполнить reboot либо `x2x reload` после применения active configuration.
4. Проверить имя, SP/ME/TF.
5. Проверить рост `success`.
6. Изменить реальные входы или выходы.
7. Проверить соответствие диагностических данных.
8. Отключить линию.
9. Проверить `connection = lost`.
10. Восстановить линию.
11. Проверить автоматическое восстановление.
12. Выполнить `rtos` до и после длительного теста.
13. Описать register map и word order в документации соответствующего модуля.

<a id="troubleshooting"></a>

## 17. Диагностика и поиск неисправностей

### 17.1 Значения Modbus RTU result

```text
idle             запрос отсутствует
busy             транзакция выполняется
ok               получен и проверен корректный ответ
timeout          ответ не получен в установленный срок
CRC error        CRC ответа не совпадает
invalid response адрес, функция, длина или данные ответа неверны
exception        slave вернул Modbus exception
transport error  ошибка передачи UART или незавершённый TX
invalid argument параметры запуска запроса недопустимы
```

### 17.2 `request = busy` и `last_result = ok`

Такое сочетание является штатным:

- предыдущий запрос завершился успешно;
- следующий запрос уже запущен.

### 17.3 `consecutive_failures = 255`

Счётчик насыщается на 255 и не переполняется. Значение указывает на длительное отсутствие корректных ответов после последнего успешного обмена.

### 17.4 Значения присутствуют, но `valid = no`

Прошивка сохраняет last-good data. Прикладное программное обеспечение обязано учитывать `valid` и `connection_lost`.

### 17.5 X2X exception code 2

Обычно соответствует `Illegal Data Address`.

Проверить:

1. ID в `X2X.TXT`;
2. версию платы и firmware модуля;
3. start address драйвера;
4. register count;
5. фактическую register map.

### 17.6 Ethernet `link = down`

Проверить:

- кабель;
- коммутатор;
- питание W5500;
- правильный разъём ETH1/ETH2;
- `VERSIONR = 0x04`;
- socket state после подключения;
- сетевые параметры компьютера.

### 17.7 microSD file not found

Проверить:

- файл находится в корне карты;
- имя полностью совпадает с каноническим;
- регистр символов;
- расширение `.TXT` или `.txt`;
- FAT16/FAT32;
- отсутствие двойного расширения;
- готовность filesystem в команде `sd`.

### 17.8 Неожиданный reset

Выполнить:

```text
watchdog
uptime
```

Проверить reset cause, boot count и время работы после последнего запуска.

### 17.9 Подозрение на утечку памяти

Повторить несколько одинаковых циклов:

```text
status
rtos
```

Сравнить:

- current free heap;
- minimum-ever-free heap;
- task stack high-water mark;
- linker-assigned SRAM.

Current free heap не должен уменьшаться после каждого одинакового цикла.

### 17.10 Консоль не принимает команды

Проверить:

- выбран правильный USB CDC-порт;
- команда завершается Enter;
- `periodic on` не создаёт непрерывный большой поток;
- контроллер не находится в цикле перезагрузки;
- uptime продолжает увеличиваться;
- после переподключения открыт актуальный COM-порт.

### 17.11 HMI не возвращает диагностический кадр

Выполнить:

```text
sc16is
```

Проверить:

- HMI обозначен как `present = yes`;
- обнаружена ожидаемая аппаратная конфигурация SC16IS;
- терминал или тестовый прибор настроен на `9600 8N1`;
- линии RS-485 подключены с правильной полярностью;
- кадр не превышает 128 байт;
- HMI не передан другому прикладному сервису;
- SPI и chip select SC16IS работают корректно.

### 17.12 X2X сначала возвращает exception, затем работает

Если после подключения модуля наблюдается временный exception, а затем:

```text
connection = online
communication_error = ok
success увеличивается
```

необходимо проверить последовательность запуска модуля и момент готовности его register map. Единичный стартовый exception следует анализировать отдельно от устойчивого потока ошибок.

### 17.13 Контроллер работает, но `source = defaults`

Причины:

- оба A/B-слота пусты или невалидны;
- выполнено `config erase confirm`;
- программатор выполнил полное стирание конфигурационной области;
- первоначальный SD import не был выполнен.

Проверить:

```text
config status
config show all
```

Это не является ошибкой нового контроллера. Для постоянной конфигурации выполнить USB provisioning либо `config import sd`.

### 17.14 SD видна, но полный импорт не выполняется

Проверить:

```text
sd
config scan sd
config status
```

Комплект должен содержать все 12 канонических файлов, пройти semantic validation и дать два полностью одинаковых чтения. После исправления файлов выполнить `config import sd`.

Если карта была установлена уже после неудачной автоматической попытки на старте, следует использовать явную команду импорта.

### 17.15 Python-команда не получает ответ

1. Закрыть PuTTY и другие программы, удерживающие COM-порт.
2. Проверить актуальный номер COM после USB reconnect.
3. Проверить наличие `pyserial`.
4. Запускать `tools/lcp_config_usb.py`, а не внутренний core module.
5. Сначала повторить `info`.
6. Убедиться, что контроллер использует firmware 1.02.0 и protocol version 1.

Утилита сама выполняет ограниченные повторы открытия Windows CDC и `HELLO`.

### 17.16 После USB-записи COM-порт исчез

Это штатный reboot изменённой конфигурации. Утилита должна дождаться повторного появления устройства и вывести:

```text
reconnected and verified
```

Если host потерял связь в неопределённый момент, нельзя вслепую повторять `PUT_CONFIG`. Сначала требуется выполнить `GET_CONFIG` или команду `read` и сравнить активный bundle.

### 17.17 Повторная запись возвращает `no change`

Это желательное поведение. Активный bundle уже совпадает с переданным, поэтому Flash и sequence не изменяются.

### 17.18 Ethernet-адреса defaults конфликтуют

Встроенные defaults одинаковы на всех новых контроллерах. Несколько устройств с одинаковыми MAC/IP нельзя одновременно подключать к одной LAN. Сначала назначить каждому контроллеру уникальную конфигурацию на изолированном подключении.

## 18. Правила разработки

### 18.1 Неблокирующая работа

Основная задача обслуживает все подсистемы. Не допускаются:

```text
delay в прикладном сервисе
busy-wait
while до аппаратного события
ожидание ответа без timeout
неограниченный цикл обработки входного буфера
```

Длительная операция должна иметь:

- состояние;
- функцию запуска;
- функцию `poll()`;
- timeout;
- явный код результата;
- определённый переход после ошибки.

### 18.2 Память

Рекомендуется использовать:

- статические буферы фиксированного размера;
- compile-time `static_assert`;
- проверку границ до записи;
- отсутствие динамического выделения в runtime;
- last-good data отдельно от quality;
- один явно определённый владелец большого общего буфера.

### 18.3 Обработка ошибок

Каждый protocol service должен:

- различать timeout, CRC, invalid response, exception и transport error;
- сохранять exception code;
- иметь порог потери связи;
- автоматически восстанавливать `online` после корректного ответа;
- не обнулять last-good data без необходимости;
- предоставлять диагностические счётчики.

### 18.4 Стиль C/C++

Настройки находятся в:

```text
.clang-format
.editorconfig
```

Основные правила:

- отступ 4 пробела;
- tab не используется;
- Allman braces;
- пробелы вокруг бинарных операторов;
- одна операция или логический шаг на строку;
- длина строки исходника до 100 символов, когда это возможно;
- понятные имена state и result;
- Doxygen для публичных интерфейсов и точек расширения.

### 18.5 Стиль диагностического вывода

```text
name = value
name = value, next = value
0: value, 1: value
```

Не следует создавать новый формат отдельно в каждом модуле. Для форматирования используются общие helpers из `diagnostic_output.hpp`.

### 18.6 Изменение архитектуры

Перед изменением общего protocol engine необходимо проверить, может ли задача быть решена на уровне конкретного service или драйвера.

Новый X2X-модуль обычно не требует изменений:

```text
x2x_config.cpp
x2x_registry.cpp
x2x_service.cpp
x2x_status.cpp
modbus_rtu_master.cpp
```

Новый внешний прибор не должен добавлять знание о своей модели в `ModbusRtuMaster`.

> ⚠️ **Внимание**
>
> Изменение универсального protocol layer затрагивает все сервисы, которые его используют. Такие изменения требуют отдельной регрессионной проверки существующих X2X-модулей и FieldSensor-портов.

### 18.7 Совместимость конфигурации и протокола

При изменении `LcpConfigBundle`, USB-протокола или storage format необходимо:

- увеличить соответствующую version;
- сохранить однозначный byte order и размеры;
- обновить firmware validation;
- обновить reference utility pack/unpack;
- проверить чтение старых слотов либо сознательно зафиксировать несовместимость;
- повторить тесты незавершённой записи, CRC, `no change`, reboot и reconnect;
- обновить формальное описание в настоящем README.

Не следует добавлять каждый прикладной параметр в schema-v1. Платформенная конфигурация должна оставаться компактной и стабильной.

- S1–S4 присутствуют и имеют ожидаемые serial parameters;
- подключённый slave переходит в `online`;
- `success` увеличивается;
- данные соответствуют внешнему прибору;
- отключение формирует `lost`;
- last-good values сохраняются, а `valid` сбрасывается;
- восстановление выполняется автоматически;
- `field pause`, `resume` и `reload` не обрывают активный кадр;
- hardware UART errors не растут без физической причины.

- quality registers соответствуют FieldSensor;
- неверная функция и адрес возвращают Modbus exception;
- `requests`, `responses`, `malformed` и `transport_errors` интерпретируются корректно;
- после disconnect/reconnect сервер восстанавливается;
- после `eth reload` клиент подключается повторно.

### 19.10 X2X regression

- active bundle создаёт ожидаемый registry;

- timeout, CRC и exception обрабатываются;
- связь восстанавливается автоматически;
- `x2x pause`, `resume` и `reload` завершают текущую транзакцию безопасно;
- при пустом списке X2X master освобождает UART для fallback echo;

### 19.11 HMI regression

- `sc16is` обнаруживает логический канал HMI;
- аппаратная схема внешних UART распознаётся корректно;

### 19.12 SD, RTC и watchdog regression

### 20.3 Автоматизированный стенд ОТК

Автоматизированный стенд является отдельным программным продуктом и не входит в LCP Basic Firmware.

Принятая архитектура:

```text
LabVIEW
    интерфейс оператора

Python
    backend, transport, orchestration, PASS/FAIL и отчёты

LCP Basic Firmware 1.02.0
    выполнение уже существующих аппаратных и протокольных сервисов
```

Стенд работает поверх неизменной firmware `v1.02.0`:

- конфигурация передаётся бинарным USB-протоколом;
- основные проверки выполняются через реальные Modbus RTU/TCP и X2X;
- текстовая консоль используется для вспомогательной диагностики;
- parser текстового вывода привязывается к версии firmware 1.02.0;
- схемы стенда, тестовые профили, алгоритмы PASS/FAIL и протоколы испытаний хранятся отдельно.

В базовую прошивку для этого не требуется добавлять bridge, `test start` или производственную логику.

### 20.4 Правило сопровождения

Изменения базовой прошивки после `v1.02.0` должны оформляться новой версией и проходить самостоятельную регрессионную проверку. Нельзя незаметно перемещать опубликованный тег.

1. сверить описание с фактическим кодом и публичным API;
2. обновить соответствующую главу настоящего README;
3. обновить `help`, диагностические поля и примеры вывода, если интерфейс изменился;
4. обновить release checklist и регрессионный объём;
5. проверить пути файлов, таблицы, относительные ссылки, якоря и кликабельное оглавление;
6. убедиться, что контроллерный README не дублирует карты и эксплуатационные особенности отдельных модулей.
