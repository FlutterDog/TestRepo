## 1. Назначение и границы базовой прошивки

`LCP Basic Diagnostic Firmware` выполняет две функции:

1. предоставляет рабочую базовую программу для контроллера LCP2116;
2. позволяет проверить основные аппаратные ресурсы и интерфейсы контроллера.

В версии 1.02.0 реализованы:

- FreeRTOS;
- одна основная прикладная задача LCP;
- неблокирующая модель обслуживания подсистем;
- четыре независимых Modbus RTU master на S1–S4;
- Modbus RTU master внутренней шины X2X;
- два независимых Modbus TCP server на ETH1 и ETH2;
- диагностический echo-test интерфейса HMI;
- служебная консоль USB CDC;
- бинарный USB-протокол чтения и записи платформенной конфигурации;
- внутреннее A/B-хранилище конфигурации во Flash ATSAM3X8E;
- опциональный строгий импорт совместимого TXT-комплекта с microSD;
- RTC, резервная батарея и аппаратный watchdog;
- диагностика SRAM, FreeRTOS heap, task stack, UART, microSD, X2X, FieldSensor и Ethernet;
- Python-утилита для конфигурирования без SD-карты.

Базовая прошивка не содержит:

- прикладного алгоритма конкретной технологической установки;
- обязательной структуры пользовательского проекта;
- универсального USB–RS-485 bridge;
- отдельного заводского автоматического теста с формированием протокола;
- PNX Studio.

Эти функции могут разрабатываться как отдельные приложения или расширения поверх предоставленных HAL, protocol и service уровней.

## 2. Аппаратные и программные интерфейсы

Контроллер построен на ATSAM3X8E с ядром ARM Cortex-M3.

```text
X2X      внутренняя модульная шина Lorentz, RS-485
S1       пользовательский RS-485 через SC16IS7xx
S2       пользовательский RS-485 через ATSAM3X8E Serial1
S3       пользовательский RS-485 через ATSAM3X8E Serial3
S4       пользовательский RS-485 через ATSAM3X8E Serial2
HMI      универсальный RS-485 через SC16IS7xx
ETH1     Ethernet на W5500
ETH2     Ethernet на W5500
USB      USB CDC: текстовая консоль и машинный конфигурационный протокол
microSD  опциональный FAT16/FAT32-носитель совместимого TXT-комплекта
RTC      часы реального времени
WDT      аппаратный watchdog
```

Назначение интерфейсов в базовой прошивке:

```text
X2X      Modbus RTU master модульной шины
S1–S4    независимые FieldSensor Modbus RTU master
HMI      неблокирующий диагностический echo-test
ETH1     Modbus TCP server, TCP port 502
ETH2     Modbus TCP server, TCP port 502
USB      служебная консоль и конфигурирование
```

Один физический half-duplex UART должен иметь только одного владельца. Нельзя одновременно обслуживать один порт echo-test, Modbus master и другим прикладным protocol engine.

## 3. Быстрый старт

### 3.1 Запуск без SD-карты

1. Открыть проект:

   ```text
   00_LCP2116/RTOS/LCP_Basic.cppproj
   ```

2. Выбрать `Release`.
3. Собрать проект без ошибок и предупреждений.
4. Прошить LCP2116.
5. Подключить USB CDC.
6. Открыть терминал и выполнить:

   ```text
   version
   status
   rtos
   config status
   field
   x2x
   eth
   sd
   time
   watchdog
   ```

Ожидаемая версия:

```text
version = 1.02.0
stage = Release 1.02.0
target = ATSAM3X8E
```

При пустой конфигурационной Flash и отсутствии корректной SD контроллер автоматически запускается на встроенных runtime defaults.

### 3.2 Запись конфигурации через USB

На компьютере должны быть установлены Python 3.9+ и `pyserial`:

```powershell
python -m pip install pyserial
```

Проверка связи:

```powershell
cd D:\IGAS_Transfer\TestRepo\00_LCP2116\RTOS
python .\tools\lcp_config_usb.py info --port COM10
```

Чтение активной конфигурации:

```powershell
New-Item -ItemType Directory -Force .\test_output | Out-Null
python .\tools\lcp_config_usb.py read `
    --port COM10 `
    --output .\test_output\active_config.json
```

Номер COM-порта зависит от конкретного компьютера.

<a id="architecture"></a>

## 4. Архитектура программного обеспечения

### 4.1 Структура проекта

```text
main.cpp
    точка входа и запуск FreeRTOS

app/
    прикладные сервисы, конфигурация, runtime и диагностика

board/
    привязка логических интерфейсов к ресурсам платы LCP2116

hal/
    ATSAM3X8E, SC16IS7xx, W5500 и внутренняя Flash

protocol/
    универсальные реализации Modbus RTU и Modbus TCP

platform/
    Serial, SPI, GPIO, millis и другие platform-функции

libs/
    независимые библиотеки, включая SD storage и CRC32

tools/
    Python-утилита конфигурирования через USB

sd_card/
    совместимый пример TXT-комплекта
```

### 4.2 Разделение уровней

```text
HAL       физические операции и регистры аппаратуры
board     аппаратная карта конкретной платы
protocol  универсальный protocol engine
service   применение протокола для конкретной функции
status    диагностика и сервисные команды
```

Пример потока FieldSensor:

```text
UART / SC16IS
    -> board transport
        -> ModbusRtuMaster
            -> FieldSensor service
                -> USB diagnostics
                -> Modbus TCP holding map
```

Пример потока X2X:

```text
UART0
    -> ModbusRtuMaster
        -> X2X service
            -> driver типа модуля
                -> runtime модуля
                    -> USB diagnostics
```

### 4.3 Правила зависимости

- Modbus engine не должен знать модель конкретного прибора.
- W5500 HAL не должен знать прикладную карту регистров.
- Драйвер X2X-модуля не должен самостоятельно читать microSD.
- Прикладной сервис не должен напрямую управлять регистрами MCU, если существует HAL или board layer.
- Длительная операция должна быть оформлена как `init()` + неблокирующий `poll()`.
- Диагностика использует публичный API сервиса и не меняет его внутренние структуры напрямую.

<a id="freertos"></a>

## 5. Модель исполнения FreeRTOS

В базовой прошивке создаётся одна прикладная задача LCP. Она последовательно вызывает короткие неблокирующие `poll()`-функции всех активных подсистем и затем отдаёт процессор планировщику.

Такой подход выбран намеренно:

- отсутствует конкуренция нескольких задач за один UART;
- нет необходимости в mutex для основных прикладных структур;
- порядок обслуживания интерфейсов детерминирован;
- диагностика отражает единое согласованное runtime-состояние;
- добавление нового сервиса не требует отдельной задачи без необходимости.

Не допускаются:

- `delay()` в прикладном обслуживании;
- busy-wait аппаратного события;
- бесконечное ожидание ответа UART, SPI или сети;
- блокирующая запись большого массива;
- захват интерфейса несколькими сервисами.

Команда:

```text
rtos
```

показывает:

- назначенную linker-ом SRAM;
- FreeRTOS heap и минимальный остаток;
- полный размер stack задачи LCP;
- максимальное фактическое использование stack.

## 6. Платформенная конфигурация

Версия schema-v1 хранит только параметры базовой платформы:

```text
4 x baudrate S1–S4
4 x parity S1–S4
ETH1 MAC, IP, subnet, gateway
ETH2 MAC, IP, subnet, gateway
список до 6 X2X-модулей
4 совместимых значения EIA
зарезервированное место для развития схемы
```

Нормализованный бинарный `LcpConfigBundle` имеет фиксированный размер 104 байта.

Приоритет источников при запуске:

```text
1. Последний корректный слот внутренней Flash.
2. Автоматический импорт с SD, только если валидных Flash-слотов нет.
3. Встроенные runtime defaults, если постоянная конфигурация отсутствует.
```

Существующая внутренняя конфигурация не заменяется автоматически отличающимся комплектом на SD. Для обновления требуется явная команда или USB-запись.

Платформенная конфигурация не определяет формат пользовательских рецептов, алгоритмов, уставок, карт сигналов или PNX-проекта. Разработчик может хранить такие данные в исходном коде, собственном бинарном формате, FRAM, SD, Ethernet-хранилище или другом выбранном источнике.

<a id="defaults"></a>

## 7. Встроенные настройки по умолчанию

При отсутствии валидной конфигурации применяются:

```text
S1 = 9600 8N1
S2 = 9600 8N1
S3 = 9600 8N1
S4 = 9600 8N1

ETH1 MAC = 02:4C:43:50:00:01
ETH1 IP = 192.168.1.1
ETH1 subnet = 255.255.255.0
ETH1 gateway = 192.168.1.254

ETH2 MAC = 02:4C:43:50:00:02
ETH2 IP = 192.168.1.2
ETH2 subnet = 255.255.255.0
ETH2 gateway = 192.168.1.254

X2X modules = none
EIA = 3, 4, 5, 6
```

Контроллер ждёт до 2 секунд возможного первоначального импорта, затем запускает S1–S4, Ethernet и X2X на активном bundle.

Defaults являются runtime-профилем и автоматически во Flash не записываются. Нормальное состояние нового контроллера:

```text
source = defaults
slot = none
sequence = 0
```

Команда:

```text
config defaults
```

только печатает встроенные значения. Она не меняет активную конфигурацию и не программирует Flash.

Одинаковые default IP и MAC предназначены для одиночного контроллера на изолированном стенде. Перед подключением нескольких новых контроллеров к одной Ethernet-сети каждому устройству необходимо назначить уникальные адреса.

<a id="flash-storage"></a>

## 8. Внутренняя Flash и слоты A/B

Последние 32 KiB внутренней Flash исключены linker script из области программы:

```text
0x000F8000 ... 0x000FFFFF
```

Область разделена на два слота:

```text
Slot A = 16 KiB
Slot B = 16 KiB
```

Одна запись включает:

1. record header;
2. нормализованный bundle;
3. CRC32 payload;
4. отдельную commit page, записываемую последней;
5. read-back verification.

Новая конфигурация всегда записывается в неактивный слот. При запуске прошивка проверяет оба слота и выбирает валидный слот с наиболее новым `sequence`.

Если запись оборвалась до корректной commit page, незавершённый слот не выбирается, а предыдущий валидный слот остаётся доступным.

Повторная передача полностью совпадающего bundle возвращает:

```text
no change
```

При этом:

- Flash не переписывается;
- sequence не увеличивается;
- reboot не выполняется.

Основные команды:

```text
config status
config rollback
config erase confirm
```

`config rollback` записывает содержимое предыдущего валидного слота как новое поколение. `config erase confirm` инвалидирует оба слота и переводит текущий runtime на defaults.

Полное стирание микроконтроллера программатором также стирает зарезервированную область конфигурации. После такого программирования контроллер запускается на defaults либо выполняет первоначальный импорт с корректной SD.

## 9. Настройка через USB

### 9.1 Назначение

USB CDC используется одновременно для:

- человекочитаемой служебной консоли;
- машинного бинарного конфигурационного протокола.

Машинный кадр начинается сигнатурой:

```text
00 4C 43 50
```

Текстовая консоль не получает байты активной машинной сессии.

Поддерживаемые protocol commands:

```text
HELLO
GET_CONFIG
VALIDATE_CONFIG
PUT_CONFIG
GET_STATUS
REBOOT
EXIT
```

Protocol version и configuration schema version независимы.

### 9.2 Python-утилита

Пользовательская точка входа:

```text
tools/lcp_config_usb.py
```

Внутренний модуль реализации schema и протокола:

```text
tools/_lcp_config_usb_core.py
```

Пользователь не должен запускать внутренний модуль напрямую.

Доступные команды:

```text
info        информация о протоколе и состоянии writer
read        чтение активной конфигурации из контроллера
write       запись JSON-модели
write-txt   импорт совместимого TXT-каталога и запись
import-txt  преобразование TXT-каталога в JSON без контроллера
export-txt  экспорт JSON в канонический TXT-комплект
```

JSON является удобным форматом этой reference utility. Он не является обязательным форматом прикладного проекта LCP.

### 9.3 Примеры PowerShell

Проверка протокола:

```powershell
python .\tools\lcp_config_usb.py info --port COM10
```

Чтение:

```powershell
python .\tools\lcp_config_usb.py read `
    --port COM10 `
    --output .\test_output\active_config.json
```

Запись JSON:

```powershell
python .\tools\lcp_config_usb.py write `
    --port COM10 `
    --input .\my_config.json
```

Запись legacy TXT:

```powershell
python .\tools\lcp_config_usb.py write-txt `
    --port COM10 `
    --directory D:\LCP_Config_USB_Test
```

Проверка TXT без контроллера:

```powershell
python .\tools\lcp_config_usb.py import-txt `
    --directory D:\LCP_Config_USB_Test `
    --output .\test_output\imported_config.json
```

Экспорт обратно в TXT:

```powershell
python .\tools\lcp_config_usb.py export-txt `
    --input .\test_output\active_config.json `
    --directory .\test_output\exported_txt
```

### 9.4 Reboot и Windows CDC

После изменённой записи контроллер должен перезапуститься, чтобы штатный config service выбрал новый слот до инициализации runtime-сервисов.

Утилита автоматически:

1. запоминает VID, PID и serial USB-устройства;
2. отправляет одну команду `REBOOT`;
3. ждёт удаления старого COM-устройства;
4. ищет контроллер после повторного появления;
5. выполняет `HELLO`;
6. читает активный bundle;
7. сравнивает его с отправленными данными.

Успешное завершение изменённой записи выглядит так:

```text
reboot required
reconnected and verified: source=1 slot=... sequence=... crc32=0x........
```

Опция:

```text
--no-reboot
```

записывает новый неактивный слот, но оставляет текущий runtime без изменений до следующего reset или отключения питания.

## 10. Опциональный импорт с microSD

microSD не требуется для обычного запуска и не является рабочим источником после успешного импорта. Активная конфигурация хранится во внутренней Flash.

Совместимый комплект состоит из 12 файлов в корне FAT16/FAT32:

```text
baud.TXT
Parity.TXT
MAC.txt
IP.txt
SUBNET.txt
GATE.txt
MAC2.txt
IP2.txt
SUBNET2.txt
GATE2.txt
X2X.TXT
EIA.TXT
```

Каждый числовой файл содержит:

```text
COUNT
VALUE_1
VALUE_2
...
fin
```

Комплект принимается только целиком. Прошивка:

1. проверяет наличие всех обязательных файлов;
2. проверяет count и диапазоны;
3. формирует нормализованный bundle;
4. читает весь комплект второй раз;
5. сравнивает оба результата и CRC;
6. только после полного совпадения начинает запись Flash.

Если Flash пустая и корректная SD доступна при запуске, выполняется автоматический первоначальный импорт.

Если на установленной карте нет полного корректного комплекта, контроллер продолжает работу на defaults. После исправления файлов рекомендуется явно выполнить:

```text
config import sd
```

Команды:

```text
config scan sd       проверить полный комплект без записи
config compare sd    сравнить SD с активной конфигурацией
config import sd     записать проверенный комплект во Flash
config show all      вывести активный bundle как канонический TXT
```

Наличие на SD отличающихся файлов не заменяет существующую Flash-конфигурацию автоматически.

## 11. Служебная консоль USB

Команды регистронезависимы и выполняются после Enter.

Основные группы:

```text
help | ?
version | ver
status
uptime
rtos

config status
config show [FILE|all]
config scan sd
config compare sd
config import sd
config rollback
config erase confirm
config defaults

field
field reload
field pause
field resume

x2x
x2x reload
x2x pause
x2x resume

eth
eth reload

rs485
sc16is
sd
sd test
battery
time
rtc
rtc set YYYY-MM-DD HH:MM:SS
watchdog
watchdog feed
watchdog test reset

periodic on
periodic off
```

`status` выводит общий отчёт. `periodic on` повторяет его каждые 10 секунд.

Некоторые команды могут относиться к демонстрационным драйверам конкретных X2X-модулей. Они являются примерами расширения и не заменяют документацию соответствующего модуля.

<a id="field"></a>

## 12. Порты S1–S4 и пример FieldSensor

Все четыре порта работают как независимые Modbus RTU master. Текущий пример по умолчанию выполняет:

```text
slave address = 1
function = FC03 Read Holding Registers
start register = 0
register count = 2
poll period = 300 ms
timeout = 500 ms
connection-loss threshold = 5 consecutive failures
```

Физическая карта:

```text
S1 -> SC16IS7xx
S2 -> Serial1
S3 -> Serial3
S4 -> Serial2
```

Каждый порт имеет собственные:

- объект `ModbusRtuMaster`;
- RX/TX-буферы;
- таймер запроса;
- timeout;
- два регистра результата;
- quality и connection state;
- счётчики успешных и неуспешных транзакций;
- время последнего успешного обновления.

После пяти последовательных ошибок:

```text
connection = lost
valid = no
```

Первый корректный ответ восстанавливает online state. Последнее полученное значение нельзя использовать без проверки `valid`.

`field reload` переинициализирует UART только после завершения активных RTU-транзакций. `field pause` запрещает новые запросы, но не обрывает уже начатый кадр.

Тип прибора, slave address, регистры и прикладное декодирование задаются в коде `field_sensor_service`. Платформенная конфигурация меняет только baudrate и parity портов.

<a id="ethernet"></a>

## 13. Ethernet и Modbus TCP

ETH1 и ETH2 реализованы на отдельных W5500 и работают как независимые Modbus TCP server:

```text
TCP port = 502
supported request = FC03 Read Holding Registers
```

Текущая read-only holding map публикует по три регистра на каждый S1–S4:

```text
value 0
value 1
quality
```

Всего используются адреса `0..11`.

Перед применением сетевых параметров проверяются:

- MAC не равен нулю и не является multicast;
- IP не равен `0.0.0.0`, multicast или limited broadcast;
- subnet mask непрерывна и не равна нулю.

Команда:

```text
eth reload
```

повторно применяет активный bundle и выполняет reset обоих W5500.

При разработке собственной карты регистров изменяются application service и документация карты. W5500 HAL и универсальный Modbus TCP engine не должны зависеть от прикладного содержания регистров.

<a id="x2x"></a>

## 14. Внутренняя шина X2X

X2X использует выделенный RS-485 интерфейс контроллера и Modbus RTU master. Состав линии задаётся в активной платформенной конфигурации.

Schema-v1 поддерживает до шести модулей. Для каждого типа требуется:

1. descriptor в статическом каталоге;
2. проверка допустимости в X2X-конфигурации;
3. driver с неблокирующей state machine;
4. регистрация runtime;
5. профильная диагностика;
6. собственная документация модуля.

Если список X2X пуст, порт не занят опросом модулей и может использовать предусмотренный в базовой прошивке fallback diagnostic echo.

`x2x reload` повторно применяет текущий активный bundle. Изменение Flash-конфигурации через USB применяется штатно после reboot.

Контроллерный README не является источником полной карты регистров отдельных модулей. Карты, алгоритмы и эксплуатационные особенности должны находиться в репозитории соответствующего устройства.

<a id="hmi"></a>

## 15. Универсальный интерфейс HMI

HMI является отдельным пользовательским RS-485 интерфейсом через SC16IS7xx. Название отражает первоначальное назначение, но программно порт может быть передан другому прикладному сервису.

В базовой прошивке работает неблокирующий echo-test:

```text
baudrate = 9600
frame = 8N1
interframe gap = 5 ms
response delay = 10 ms
maximum frame = 128 bytes
TX timeout = 1000 ms
```

Поддерживаются варианты SC16IS740 и SC16IS752. Аппаратная карта определяется при запуске.

Для передачи HMI пользовательскому протоколу необходимо:

1. исключить HMI из echo-test;
2. получить descriptor порта из board layer;
3. создать отдельный неблокирующий service;
4. реализовать timeout, частичный TX и очистку RX;
5. добавить профильную диагностику;
6. гарантировать единственного владельца UART.

<a id="extension"></a>

## 16. Добавление прикладных функций

### 16.1 Новый Modbus RTU-прибор

Для простого прибора можно изменить текущий FieldSensor example. Для сложного устройства создаётся отдельный service поверх `ModbusRtuMaster`.

Отдельный service рекомендуется, если требуются:

- несколько последовательных запросов;
- операции записи;
- собственная state machine;
- преобразование нескольких типов данных;
- отдельные quality flags;
- собственная карта публикации;
- независимая диагностика.

При увеличении числа регистров необходимо согласованно изменить runtime buffer, request parameters, декодирование, status и Ethernet map.

### 16.2 Новый X2X-модуль

Новый модуль добавляется через каталог и driver. Универсальный X2X transport и Modbus RTU engine изменять не требуется, если протокол линии остаётся совместимым.

### 16.3 Новая команда консоли

Команда должна:

- завершаться быстро;
- не ожидать аппаратное событие в цикле;
- запускать длительную операцию через service API;
- иметь строку в `help`;
- выдавать однозначный status;
- не раскрывать внутренние mutable structures сервиса.

### 16.4 Пользовательская конфигурация

Разработчик не обязан расширять schema-v1 для каждого прикладного параметра. Возможны:

- hardcoded параметры программы;
- собственный бинарный Flash-format;
- FRAM;
- SD-файлы любого выбранного формата;
- данные по Ethernet;
- отдельная project database.

Schema-v1 предназначена для базовых аппаратных параметров LCP2116.

<a id="troubleshooting"></a>

## 18. Диагностика и поиск неисправностей

### Контроллер работает, но source = defaults

Причина: нет валидного Flash-слота или выполнено полное стирание.

Проверка:

```text
config status
config show all
```

Решение: записать конфигурацию через USB либо выполнить `config import sd`.

### USB Python-команда не получает ответ

1. Закрыть PuTTY и другие программы, удерживающие COM-порт.
2. Проверить номер COM в диспетчере устройств.
3. Проверить `pyserial`.
4. Повторить `info`.
5. Проверить, что используется `lcp_config_usb.py`, а не внутренний core module.

Утилита сама повторяет открытие Windows CDC и HELLO в пределах ограниченного timeout.

### После записи COM-порт исчез

Это нормальный reboot изменённой конфигурации. Утилита должна дождаться повторного появления и вывести `reconnected and verified`.

### Повторная запись возвращает no change

Это нормальное и желательное поведение. Flash и sequence не изменяются.

### SD видна, но импорт не выполняется

Проверить:

```text
sd
config scan sd
config status
```

Комплект должен содержать все 12 файлов и пройти два одинаковых чтения. После исправления файлов выполнить `config import sd`.

### S1–S4 не переходят online

Проверить:

- slave address;
- baudrate и parity;
- A/B и землю RS-485;
- направление подключения;
- наличие terminating resistor при необходимости;
- `rs485` и UART error counters;
- `field` и consecutive failures;
- ожидаемые FC03 registers внешнего slave.

### Ethernet не отвечает

Проверить:

- link LED;
- активный IP и subnet через `eth`;
- отсутствие конфликта IP/MAC;
- соединение с правильным ETH1/ETH2;
- TCP port 502;
- reset и init status W5500.
