# Проверка базовой подсистемы X2X

Статус: **реализация добавлена в `feature/freertos-baseline`, аппаратные испытания не выполнены**.

Документ описывает проверку первой переносимой версии подсистемы:

```text
X2X.TXT
→ проверка конфигурации
→ статический реестр модулей
→ Modbus RTU master UART0
→ специализированные драйверы LDO/LAI/LDI/LCT
→ USB-диагностика
```

## 1. Предварительные условия

- контроллер LCP2116 с ATSAM3X8E;
- microSD с FAT16/FAT32;
- USB CDC подключён к терминалу 115200 8N1;
- X2X работает на UART0, 9600 8N1;
- модули имеют последовательные адреса 1..N в порядке, указанном в `X2X.TXT`;
- максимальное количество внешних X2X-модулей в данной версии — 6.

## 2. Стабильные ID модулей

```text
0 — LCP2116, в X2X.TXT запрещён
1 — LDO1118
2 — LAI1118
3 — LDI1118, 8 дискретных входов
4 — LCT1114
5 — LDI1116, 16 дискретных входов
6 — LCT1114_2
```

Назначенные значения ID являются частью формата конфигурации и не должны изменяться в последующих версиях.

## 3. Базовый X2X.TXT

Пример для четырёх семейств модулей:

```text
4
1 // slave 1: LDO1118
2 // slave 2: LAI1118
5 // slave 3: LDI1116
6 // slave 4: LCT1114_2
Fin
```

Строка `Fin` необязательна. Разрешены пустые строки и комментарии `//`.

## 4. Проверка сборки

Собрать последовательно:

```text
Debug
Release
```

Проверить:

- все новые `.cpp` действительно компилируются;
- в Debug и Release найден `FreeRTOS.h`;
- отсутствуют undefined reference для X2X и Modbus RTU;
- итоговый размер RAM не превышает 96 KiB;
- итоговый размер Flash не превышает 512 KiB;
- нет duplicate symbol для UART handlers, SysTick, PendSV и SVC.

Зафиксировать из отчёта линкера:

```text
text=
data=
bss=
```

## 5. Проверка загрузчика X2X.TXT

После каждого изменения файла выполнить:

```text
x2x reload
x2x
```

### 5.1 Корректный файл

Ожидается:

```text
x2x reload: ok
loaded=yes
registry=ok
modules=4
```

Адреса и ID должны совпадать с порядком строк.

### 5.2 Пустая конфигурация

```text
0
Fin
```

Ожидается:

- конфигурация принята;
- `modules=0`;
- X2X UART возвращён диагностическому echo-test;
- опрос не запускается.

### 5.3 Отсутствующий файл

Удалить `X2X.TXT`.

Ожидается:

```text
result=file not found
loaded=no
```

Если до команды `reload` работала корректная конфигурация, она должна продолжить работать. После перезапуска без файла активной конфигурации быть не должно.

### 5.4 Запрещённый LCP

```text
1
0
```

Ожидается:

```text
result=invalid X2X device ID
config_error_line=2
value=0
```

### 5.5 Неизвестный ID

```text
1
7
```

Ожидается ошибка ID на второй физической строке.

### 5.6 Слишком много модулей

```text
7
1
1
1
1
1
1
1
```

Ожидается `invalid module count` на первой строке.

### 5.7 Обрезанный файл

```text
4
1
2
```

Ожидается `incomplete file`.

### 5.8 Лишние данные

```text
1
2
5
```

Ожидается `extra data` на строке с ID 5.

### 5.9 Комментарии и пустые строки

```text
// X2X configuration

2

2 // LAI
5 // LDI16

Fin
```

Ожидается успешная загрузка двух модулей.

## 6. Проверка владения UART0

При корректной конфигурации с `N > 0` команда `rs485` должна показать:

```text
X2X ... mode=Modbus RTU master, echo=disabled
```

При конфигурации `0` или при отсутствии корректного файла:

```text
X2X ... mode=echo test, echo=enabled
```

Порты S2, S3 и S4 должны продолжить echo-test независимо от X2X.

## 7. Проверка LDI1118 и LDI1116

Для каждого модуля:

1. Установить его единственным устройством в `X2X.TXT`.
2. Проверить правильный физический slave address 1.
3. Выполнить `x2x`.
4. Переключать каждый дискретный вход.
5. Контролировать изменение `digital_inputs`.

Ожидаемые маски:

```text
LDI1118: используются биты 0..7
LDI1116: используются биты 0..15
```

После успешного ответа:

```text
connection=online
success увеличивается
consecutive_failures=0
```

## 8. Проверка LAI1118

Конфигурация:

```text
1
2
```

Проверить:

- читается 17 holding-регистров с адреса 0;
- регистр 0 обновляет `digital_inputs`;
- регистры 1..16 формируют 8 значений float;
- порядок слов совпадает со старой прошивкой: младшее 16-битное слово первым, старшее вторым;
- значения `float_values 0..7` соответствуют модулю.

Сравнить минимум две контрольные величины с известным значением тока или с Modbus-анализатором.

## 9. Проверка LDO1118

Конфигурация:

```text
1
1
```

Команды:

```text
x2x ldo 1 0
x2x ldo 1 1
x2x ldo 1 255
```

Проверить:

- функция 0x10 пишет один регистр с адреса 0;
- физические выходы соответствуют битовой маске;
- ответ slave подтверждает тот же адрес и количество регистров;
- отрицательное значение передаётся как 16-битное дополнительное представление только для сервисной проверки и не должно применяться без необходимости.

## 10. Проверка LCT1114

### 10.1 Основные данные

Драйвер сохраняет запрос старой прошивки:

```text
start=0
count=78
```

Из первых двух регистров формируется 19-битная карта DI. Следующие регистры содержат float.

⚠️ В старом коде структура объявляет 42 float-канала, но запрос 78 регистров содержит только 38 float после двух DI-регистров. В переносимой версии:

- декодируются 38 фактически доступных float;
- оставшиеся 4 поля обнуляются;
- изменение количества регистров запрещено до подтверждения реальной карты LCT1114.

Во время проверки необходимо снять полный ответ модуля или сверить карту регистров и решить, должен ли основной запрос содержать 78 либо 86 регистров.

### 10.2 Осциллограммы

Проверить последовательность:

```text
read 850, count 1
если значение 1:
    read 200..399, 200 регистров
    read 400..599, 200 регистров
    read 600..799, 200 регистров
    write 0 в регистр 850
```

Каждый диапазон 200 регистров должен физически разбиваться на два запроса по 100 без запроса с количеством 0.

После полного чтения:

```text
waveform_valid=yes
samples_per_phase=200
sequence увеличивается
```

## 11. Проверка LCT1114_2

Основной запрос:

```text
start=0
count=94
```

Должны декодироваться:

```text
2 регистра DI
46 значений float
```

Осциллограммы:

```text
201..395 — 195 регистров
401..595 — 195 регистров
601..795 — 195 регистров
```

Каждая фаза должна читаться блоками 100 + 95. После успешного сохранения повторная проверка флага откладывается примерно на 1000 мс.

## 12. Проверка ошибок и восстановления

Для каждого типа модуля:

1. Дождаться состояния `online`.
2. Разорвать X2X-линию.
3. Проверить увеличение `failed` и `consecutive_failures`.
4. После пяти последовательных ошибок должно появиться `connection=lost`.
5. Восстановить линию.
6. После первого корректного ответа ожидается:

```text
connection=online
consecutive_failures=0
communication_error=0
```

Отдельно проверить:

- неправильный CRC;
- ответ другого slave;
- неправильный function code;
- Modbus exception response;
- неполный ответ;
- отсутствие ответа.

## 13. Совместная работа baseline-функций

При активном X2X оставить систему минимум на 30 минут и проверять:

```text
periodic on
```

Не должно происходить:

- зависания USB CDC;
- остановки PLC_ok;
- ложного watchdog reset;
- зависания Ethernet;
- прекращения обслуживания S2–S4;
- роста `consecutive_failures` у исправных модулей;
- уменьшения `minimum_ever_free_heap_bytes` при каждом цикле.

Повторить:

```text
rtos
status
x2x
sd
watchdog
```

## 14. USB 1200 baud и SAM-BA

После завершения X2X-проверок открыть и закрыть USB CDC на 1200 baud.

Ожидается штатный переход в ROM SAM-BA и возможность повторной прошивки. Активный X2X не должен мешать этому механизму.
