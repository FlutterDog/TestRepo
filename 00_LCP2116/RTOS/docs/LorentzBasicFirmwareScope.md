# Границы базовой прошивки Lorentz

## Назначение

Базовая прошивка Lorentz является демонстрационной и эксплуатационной основой свободнопрограммируемого контроллера LCP2116.

Она показывает три независимых механизма:

1. работу модулей системы по внутренней шине X2X;
2. циклический опрос внешнего Modbus RTU-прибора через S1–S4;
3. публикацию данных через Modbus TCP на двух Ethernet-интерфейсах.

IEC 60870-5-104, PNX Studio, панель оператора и прикладная логика конкретной установки в эту прошивку не входят.

Готовые штатные файлы microSD находятся в `00_LCP2116/RTOS/sd_card`.

## 1. X2X

Поддерживаемые модули:

```text
LDO1118
LAI1118
LDI1118
LDI1116
LCT1114
LCT1114_2
```

Состав и порядок модулей задаются в `X2X.TXT`; позиция в файле определяет slave address.

Реализованы неблокирующий обмен, проверка timeout/CRC/exception, сохранение последнего корректного значения, отдельное состояние качества, автоматическое восстановление, команда `x2x`, статический registry и централизованный catalog драйверов.

## 2. FieldSensor на S1–S4

Демонстрационный прибор жёстко задан в исходном коде:

```text
slave address: 1
function:      0x03 Read Holding Registers
start:         0
count:         2
period:        300 ms
timeout:        500 ms
```

Один и тот же пример независимо работает на S1, S2, S3 и S4. Каждый порт имеет отдельный transport, `ModbusRtuMaster`, буферы, таймеры, значения, качество и счётчики.

После пяти последовательных ошибок устанавливаются `connection=lost` и `valid=no`. Последние значения не обнуляются. Первый корректный ответ восстанавливает связь.

### Аппаратные настройки

Из microSD читаются только канонические файлы:

```text
baud.TXT
Parity.TXT
```

`baud.TXT` содержит реальные значения скорости. `Parity.TXT` принимает только коды `0=None`, `1=Odd`, `2=Even`. Порядок значений: S1, S2, S3, S4. Значения по умолчанию — 9600 8N1.

`SC16.txt` больше не используется: вариант SC16IS740/752 определяется автоматически.

`EIA.TXT` сохраняется как штатный формат переназначения логических потоков `ch1..ch4` на физические порты, но в baseline-runtime не применяется, поскольку логических потоков здесь нет.

Команды:

```text
field
field reload
field pause
field resume
```

Минимальный reusable Modbus RTU slave FC03 входит в исходники, но пример его подключения отключён по умолчанию.

Подробности: `docs/FieldSensorExample.md`.

## 3. Ethernet Modbus TCP

Оба W5500 работают как независимые Modbus TCP server:

```text
ETH1: TCP port 502
ETH2: TCP port 502
function: 0x03 Read Holding Registers
map: 0..11
```

Сетевые параметры читаются строго из канонических файлов:

```text
ETH1: MAC.txt,  IP.txt,  SUBNET.txt,  GATE.txt
ETH2: MAC2.txt, IP2.txt, SUBNET2.txt, GATE2.txt
```

Псевдонимы и альтернативные имена не поддерживаются.

Оба интерфейса предоставляют одинаковую read-only карту:

```text
0   S1 register 0
1   S1 register 1
2   S1 quality
3   S2 register 0
4   S2 register 1
5   S2 quality
6   S3 register 0
7   S3 register 1
8   S3 quality
9   S4 register 0
10  S4 register 1
11  S4 quality
```

Команды:

```text
eth
eth reload
```

Подробности: `docs/ModbusTcpExample.md`.

## Что намеренно не входит

```text
IEC 60870-5-104
панель оператора
PNX Studio
проектные файлы PNX
база типов внешних приборов
динамическое добавление сенсоров пользователем
технологические алгоритмы конкретной установки
релейные матрицы PNX
газовые таблицы и специализированные драйверы PNX
```

## Оставшиеся действия перед релизом

1. Собрать Debug и Release после фиксации канонического формата microSD.
2. Проверить загрузку `baud.TXT` и `Parity.TXT` с завершающей строкой `fin`.
3. Проверить каждый FieldSensor-порт отдельно и совместно.
4. Проверить timeout, потерю и автоматическое восстановление.
5. Проверить загрузку MAC/IP/SUBNET/GATE для ETH1 и ETH2.
6. Проверить оба Modbus TCP server на порту 502.
7. Сравнить данные TCP 0..11 с командой `field`.
8. Проверить совместную работу X2X, FieldSensor и Ethernet.
9. Зафиксировать RAM, FreeRTOS heap и стек.
10. Выпустить финальный релиз только после явного подтверждения.
