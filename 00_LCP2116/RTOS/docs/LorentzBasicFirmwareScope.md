# Границы базовой прошивки Lorentz

## Назначение

Базовая прошивка Lorentz является демонстрационной и эксплуатационной основой свободнопрограммируемого контроллера LCP2116.

Она показывает заказчику три независимых механизма:

1. работу модулей системы по внутренней шине X2X;
2. пример циклического опроса внешнего прибора по пользовательским линиям RS-485;
3. пример обмена по Ethernet через Modbus TCP.

Проект PNX, IEC 60870-5-104, панель оператора и прикладная логика конкретной установки в эту прошивку не входят.

## 1. Модули X2X

Базовая прошивка поддерживает разработанные модули:

```text
LDO1118
LAI1118
LDI1118
LDI1116
LCT1114
LCT1114_2
```

Конфигурация состава считывается из:

```text
X2X.TXT
```

Порядок строк определяет Modbus slave address модуля.

Для каждого модуля реализованы:

- неблокирующий циклический опрос или запись;
- контроль timeout, CRC, exception и формата ответа;
- фиксация последнего корректного значения;
- отдельная метка качества через состояние связи и ошибки;
- автоматическое восстановление после возврата связи;
- диагностический вывод через команду `x2x`.

Значения при потере связи не обнуляются.

## 2. Пример внешнего прибора FieldSensor

В проект добавлен демонстрационный прибор:

```text
FieldSensor
```

Это не база приборов и не конфигурируемый список сенсоров. Прибор задаётся в исходном коде как понятный пример реализации.

Один и тот же пример запускается независимо на линиях:

```text
S1
S2
S3
S4
```

Для каждой линии выполняется:

```text
Modbus RTU function: 0x03 Read Holding Registers
slave address: 1
start register: 0
register count: 2
poll period: 300 ms
serial format: 9600 8N1
timeout: 500 ms
```

Каждая линия имеет собственные:

- аппаратный transport;
- машину состояния Modbus master;
- RX/TX-буферы;
- два последних корректных регистра;
- состояние valid/invalid;
- счётчики успешных и ошибочных запросов;
- код exception;
- время последнего корректного обновления.

Опрос S1–S4 выполняется неблокирующе внутри базовой задачи LCP. Отдельная RTOS-задача на каждый порт не создаётся.

При потере связи последние значения сохраняются. После пяти последовательных ошибок устанавливаются `connection=lost` и `valid=no`. Первый корректный ответ автоматически восстанавливает связь.

Диагностические команды:

```text
field
field pause
field resume
```

Активная конфигурация использует все четыре порта как Modbus master. Минимальный reusable slave с функцией 0x03 также включён в исходники, но пример его подключения отключён по умолчанию, поскольку один half-duplex-порт не может одновременно работать как master и slave.

Подробное описание:

```text
docs/FieldSensorExample.md
```

### Аппаратные настройки портов

Этап 17A намеренно использует ясную hardcoded-конфигурацию 9600 8N1. Это позволяет сначала проверить четыре физических транспорта и независимые автоматы Modbus master без влияния парсера microSD.

После аппаратного подтверждения параметры UART могут быть загружены из файлов microSD. Файл со списком датчиков для базовой прошивки не используется: тип `FieldSensor`, адрес и карта двух регистров остаются примером в исходном коде.

## 3. Ethernet и Modbus TCP

После аппаратной проверки FieldSensor добавляется пример Modbus TCP на Ethernet-порту LCP.

Минимальный пример:

- статический IP считывается из существующего файла сетевых настроек;
- W5500 инициализируется как Modbus TCP server;
- значения `FieldSensor` публикуются в holding registers;
- отдельные status/quality registers показывают валидность и ошибку связи;
- последние значения не заменяются нулём при потере RS-485 связи.

Предварительная карта:

```text
0      FieldSensor register 0 from S1
1      FieldSensor register 1 from S1
2      quality/status S1
3      FieldSensor register 0 from S2
4      FieldSensor register 1 from S2
5      quality/status S2
6      FieldSensor register 0 from S3
7      FieldSensor register 1 from S3
8      quality/status S3
9      FieldSensor register 0 from S4
10     FieldSensor register 1 from S4
11     quality/status S4
```

Окончательная карта фиксируется после реализации Modbus TCP.

## Что намеренно не входит

```text
IEC 60870-5-104
панель оператора
PNX Studio
проектные файлы PNX
база типов внешних приборов
динамическое добавление сенсоров пользователем
технологические алгоритмы конкретной установки
релейные матрицы PNX
газовые таблицы и специализированные драйверы PNX
```

Эти функции относятся к отдельному проекту PNX и не должны усложнять базовый комплект Lorentz.

## Порядок завершения

1. Собрать Debug и Release после добавления FieldSensor.
2. Проверить S1, S2, S3 и S4 отдельно.
3. Проверить четыре порта одновременно.
4. Проверить timeout, потерю и автоматическое восстановление связи.
5. Зафиксировать RAM, FreeRTOS heap и стек после добавления четырёх master.
6. При необходимости подключить чтение аппаратных параметров UART из microSD.
7. Реализовать Modbus TCP server и публикацию данных FieldSensor.
8. Восстановить чтение IP-настроек Ethernet из старой прошивки.
9. Провести длительный совместный тест X2X, FieldSensor и Ethernet.
10. Привести комментарии и руководства в финальный вид.
11. Выпустить финальный релиз базовой прошивки Lorentz.
