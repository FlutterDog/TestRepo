# Пример FieldSensor на S1–S4

## Назначение

`FieldSensor` — минимальный демонстрационный драйвер внешнего Modbus RTU-прибора для базовой прошивки Lorentz.

Это не база приборов и не конфигурируемый список датчиков. Тип прибора, адрес slave и карта регистров заданы в исходном коде. TXT-файлы на microSD изменяют только аппаратные параметры последовательных портов.

Готовые штатные файлы находятся в:

```text
00_LCP2116/RTOS/sd_card/
```

## Активный пример

Все четыре пользовательских порта работают как независимые Modbus RTU master:

```text
S1 -> slave 1, FC03, holding registers 0..1
S2 -> slave 1, FC03, holding registers 0..1
S3 -> slave 1, FC03, holding registers 0..1
S4 -> slave 1, FC03, holding registers 0..1
```

Общие параметры:

```text
slave address:  1
start register: 0
register count: 2
poll period:    300 ms
timeout:        500 ms
```

## Аппаратное соответствие

```text
S1 -> внешний UART SC16IS7xx
S2 -> ATSAM3X8E Serial1 / HAL UART1
S3 -> ATSAM3X8E Serial3 / HAL UART3
S4 -> ATSAM3X8E Serial2 / HAL UART2
```

Для S1 автоматически определяется SC16IS752 channel B или отдельный SC16IS740. Файл `SC16.txt` больше не используется.

## Канонические настройки microSD

Имена файлов являются частью существующего формата проектов. Псевдонимы не поддерживаются.

### baud.TXT

Первая значимая строка — количество портов. Далее идут фактические скорости в порядке S1, S2, S3, S4:

```text
4
9600
9600
1200
9600
fin
```

Это реальные значения baudrate, а не индексы таблицы.

### Parity.TXT

```text
4
0
0
0
0
fin
```

Допустимы только штатные коды:

```text
0 = None, 8N1
1 = Odd,  8O1
2 = Even, 8E1
```

Буквы `N/O/E`, слова `NONE/ODD/EVEN` и ASCII-коды не поддерживаются.

Комментарии после `//` или `#` и пустые строки игнорируются. После чтения четырёх значений строка `fin` игнорируется.

Если файл отсутствует или содержит ошибку, соответствующая группа параметров сохраняет значение по умолчанию `9600 8N1`. Скорость и чётность загружаются независимо.

## EIA.TXT

В прикладных проектах `EIA.TXT` переназначает логические потоки `ch1..ch4` на физические порты:

```text
3 = S1
4 = S2
5 = S3
6 = S4
```

Это позволяет перенести поток на другой исправный порт без перепрошивки. В базовом примере Lorentz логических потоков нет, поэтому `EIA.TXT` сохранён в комплекте, но runtime его не применяет. Каждый `FieldSensor` напрямую закреплён за S1, S2, S3 или S4.

## Применение настроек

При запуске прошивка ждёт монтирования microSD до двух секунд. Повторное чтение без перезагрузки:

```text
field reload
```

Активные транзакции завершаются, после чего все четыре UART переинициализируются.

Команда:

```text
field
```

показывает результат загрузки `baud.TXT` и `Parity.TXT`, а также фактически применённые `baud` и `format` каждого порта.

## Планирование и состояние

Отдельные FreeRTOS-задачи для портов не создаются. Один `field_sensor_service_poll()` обслуживает четыре независимых объекта `ModbusRtuMaster`.

Каждый порт имеет собственные transport, RX/TX-буферы, таймер, два регистра, состояние транзакции, признаки качества, счётчики и время последнего успешного обновления.

При единичной ошибке последние корректные значения сохраняются. После пяти последовательных ошибок устанавливаются:

```text
valid=no
connection=lost
```

Первый корректный ответ автоматически восстанавливает связь.

## Владение UART

При запуске FieldSensor echo-test отключается на S1–S4. X2X независимо использует UART0. Один half-duplex-порт не используется одновременно несколькими protocol engine.

## Service console

```text
field
field reload
field pause
field resume
```

## Альтернативный slave-режим

Минимальный reusable slave FC03 расположен в:

```text
protocol/modbus_rtu/modbus_rtu_slave.cpp
protocol/modbus_rtu/modbus_rtu_slave.hpp
```

Отключённый пример подключения:

```text
app/field/field_sensor_slave_example.cpp
```

Рабочая baseline-конфигурация оставляет все S1–S4 в режиме master.

## Проверка

1. Скопировать `baud.TXT` и `Parity.TXT` в корень microSD.
2. Выполнить `field reload` либо перезапустить контроллер.
3. Проверить результат чтения обоих файлов и параметры каждого порта.
4. Настроить внешний slave address 1 на параметры соответствующего порта.
5. Проверить `valid=yes`, `connection=online` и рост `success`.
6. Проверить потерю и автоматическое восстановление связи.
