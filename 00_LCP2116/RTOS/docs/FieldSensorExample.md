# Пример FieldSensor на S1–S4

## Назначение

`FieldSensor` — минимальный демонстрационный драйвер внешнего Modbus RTU-прибора для базовой прошивки Lorentz.

Это не база приборов и не конфигурируемый список датчиков. Пример показывает заказчику, где задаётся карта регистров, как создаётся независимый master для каждого физического порта и как обрабатываются качество и ошибки связи.

## Активная конфигурация

Все четыре пользовательских порта работают как независимые Modbus RTU master:

```text
S1 -> slave 1, FC03, holding registers 0..1
S2 -> slave 1, FC03, holding registers 0..1
S3 -> slave 1, FC03, holding registers 0..1
S4 -> slave 1, FC03, holding registers 0..1
```

Параметры baseline:

```text
baudrate:       9600
frame:          8N1
slave address:  1
start register: 0
register count: 2
poll period:    300 ms
timeout:        500 ms
```

Одинаковый адрес slave допустим, поскольку S1, S2, S3 и S4 являются четырьмя физически независимыми линиями.

## Аппаратное соответствие

```text
S1 -> обнаруженный внешний UART SC16IS7xx
S2 -> ATSAM3X8E UART/USART port 1, Serial1
S3 -> ATSAM3X8E UART/USART port 3, Serial3
S4 -> ATSAM3X8E UART/USART port 2, Serial2
```

Для S1 сохраняется автоматическое определение двух аппаратных вариантов:

```text
SC16IS752: UART2_CS, channel B
SC16IS740: UART3_CS, channel A
```

## Планирование

Отдельные FreeRTOS-задачи для портов не создаются. Один `field_sensor_service_poll()` неблокирующе обслуживает четыре независимых объекта `ModbusRtuMaster` внутри базовой задачи LCP.

Каждый порт имеет собственные:

- транспорт;
- RX/TX-буферы Modbus master;
- таймер периодического запуска;
- два регистра данных;
- признак активного запроса;
- состояние valid/invalid;
- счётчики успешных и ошибочных запросов;
- счётчик последовательных ошибок;
- код Modbus exception;
- время последнего корректного обновления.

Запросы четырёх портов могут одновременно находиться в разных состояниях ожидания ответа.

## Семантика качества

При корректном ответе:

```text
valid=yes
connection=online
consecutive_failures=0
```

При единичной ошибке последние корректные значения сохраняются.

После пяти последовательных ошибок:

```text
valid=no
connection=lost
```

Регистры при этом не обнуляются. После первого корректного ответа связь автоматически восстанавливается.

## Владение UART

При запуске FieldSensor:

- echo-test отключается на S1, S2, S3 и S4;
- echo-test PC и HMI на остальных каналах SC16IS остаётся доступным;
- X2X продолжает независимо владеть UART0, когда в `X2X.TXT` есть модули;
- при пустом `X2X.TXT` UART0 может вернуться X2X echo-test.

## Service console

Основной отчёт:

```text
field
```

Управление запуском новых запросов:

```text
field pause
field resume
```

`field pause` не обрывает активный RTU-кадр: уже начатые запросы завершаются, после чего новые не запускаются.

## Изменение примера прибора

Параметры демонстрационного прибора находятся в:

```text
app/field/field_sensor_service.cpp
```

Массив `g_default_configs` задаёт роль, адрес slave, карту регистров, период, timeout и параметры UART для каждого порта.

Для специализированного прибора заказчик может:

1. создать отдельный сервис или драйвер;
2. использовать существующий `ModbusRtuMaster`;
3. заменить адреса и количество регистров;
4. добавить декодирование данных после успешной транзакции;
5. сохранить принятую семантику качества и последнего корректного значения.

## Альтернативный slave-режим

Рабочая baseline-конфигурация использует все S1–S4 как master.

Минимальный reusable slave с поддержкой `0x03 Read Holding Registers` находится в:

```text
protocol/modbus_rtu/modbus_rtu_slave.hpp
protocol/modbus_rtu/modbus_rtu_slave.cpp
```

Отключённый пример подключения двух holding-регистров к S1 находится в:

```text
app/field/field_sensor_slave_example.cpp
```

Один физический half-duplex-порт нельзя одновременно использовать как master и slave. Для включения примера роль выбранного порта должна быть изменена на `FIELD_PORT_SLAVE`, а master-опрос этого порта исключён.

## Аппаратная проверка

### Проверка одного порта

1. Подключить Modbus slave address 1 к проверяемому порту.
2. Настроить 9600 8N1.
3. Предоставить holding-регистры 0 и 1.
4. Выполнить `field`.
5. Проверить рост `success`, значения `registers`, `valid=yes` и `connection=online`.

### Проверка потери связи

1. Разорвать линию.
2. Дождаться минимум пяти ошибок.
3. Проверить `connection=lost` и `valid=no`.
4. Убедиться, что последние значения не обнулились.
5. Восстановить линию.
6. Проверить автоматический переход в `online` после корректного ответа.

### Проверка четырёх портов

Подключить четыре независимых slave с одинаковым адресом 1 и различными значениями регистров. Команда `field` должна показывать независимые данные и счётчики S1, S2, S3 и S4.

## Настройки из microSD

Этап 17A использует явно видимую hardcoded-конфигурацию 9600 8N1, чтобы сначала проверить аппаратные транспорты и неблокирующую работу четырёх master.

После аппаратного подтверждения формат аппаратных настроек портов может быть подключён к microSD без изменения драйвера FieldSensor. Файл со списком датчиков для базовой прошивки не требуется.
