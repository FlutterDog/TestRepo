# Stage 0 — Start (`clone-first`)

## Purpose

Stage 0 выполняется один раз на новый репозиторий до загрузки реального freeze-комплекта.

Цель стадии:

* проверить, что template-репозиторий собран корректно;
* зафиксировать рабочий Board ID для проекта;
* подтвердить канонические пути и модель жизни артефактов;
* подготовить переход в Stage 1 без переименований и без «второй правды».

Stage 0 не делает freeze и не пишет содержательные артефакты Board Guide.

---

## Scope of this stage

Stage 0 отвечает только за стартовую готовность репозитория.

На этой стадии нужно проверить:

* корневую структуру репозитория;
* наличие `README.md` и `docs/authoring/pipeline/PIPELINE.md`;
* наличие stage-папок и authoring/release-зон;
* понятность будущего Board ID;
* готовность к загрузке реальных инженерных исходников на Stage 1.

---

## Operator-only статус Stage 0

Stage 0 — операторская стадия.

Для неё не требуется:

* `message_prompt.md`;
* `stage_prompt.md`;
* генерация `release_manifest.md`, `file_map.md`, `fact_pack.md` и других рабочих артефактов.

Первый канонический проектный артефакт появляется только на Stage 1.

---

## Inputs

Минимальные входы Stage 0:

* сам template-репозиторий;
* `/README.md`;
* `docs/authoring/pipeline/PIPELINE.md`.

Желательно уже знать:

* рабочее имя платы;
* ревизию;
* ожидаемый тип инженерных исходников, которые будут загружаться на Stage 1.

---

## Что Stage 0 обязан подтвердить

### Репозиторий готов к работе

Должны существовать:

* `sources/`;
* `docs/authoring/pipeline/`;
* `docs/authoring/artifacts/`;
* `docs/release/`.

### Канонические пути уже понятны

На Stage 0 достаточно подтвердить правило, а не раскладывать реальные файлы:

* инженерные исходники в дальнейшем живут только в `sources/*`;
* рабочие артефакты стадий живут только в `docs/authoring/artifacts/`;
* authoring-изображения живут только в `docs/authoring/artifacts/IMG/`;
* релизный документ и релизные изображения живут только в `docs/release/`.

### Board ID определён

На Stage 0 нужно выбрать рабочую строку Board ID, которой затем будет пользоваться весь пайплайн.

Важно:

* первое каноническое сохранение Board ID происходит в `docs/authoring/artifacts/stage_1_freeze_sources/release_manifest.md` на Stage 1;
* на Stage 0 Board ID ещё не является отдельным артефактом, а считается зафиксированным решением оператора.

---

## Что Stage 0 не делает

Stage 0 не должен:

* загружать и канонически раскладывать freeze-комплект;
* анализировать схему или PCB;
* писать `release_manifest.md`;
* писать `file_map.md`;
* писать `fact_pack.md`, `glossary.md`, `open_items.md`, `outline.md`, `draft` или release.

Если на Stage 0 уже началась раскладка реальных источников по `sources/*`, это значит, что работа фактически перешла в Stage 1.

---

## Procedure

### First) Прочитать корневые документы

Обязательно прочитать:

* `/README.md`;
* `docs/authoring/pipeline/PIPELINE.md`.

Задача этого шага — подтвердить, что оператор понимает общую логику: Release vs Authoring, жизненный цикл `sources/*`, и порядок стадий.

### Second) Проверить каркас репозитория

Проверить, что template-репозиторий уже содержит ожидаемые папки и что структура не была случайно сломана.

На этом шаге не нужно описывать каждую подпапку заново — достаточно убедиться, что каркас соответствует корневому `README` и `PIPELINE.md`.

### Third) Подтвердить канонические пути

Подтвердить три правила:

* `sources/*` — будущий Source of Truth;
* `docs/authoring/artifacts/*` — рабочая зона стадий;
* `docs/release/*` — чистый релизный выход.

Дополнительно проверить, что не планируется использовать устаревший путь `docs/authoring/IMG/`.

### Fourth) Зафиксировать рабочий Board ID

Выбрать и записать для себя рабочую форму Board ID.

Она должна быть:

* однозначной;
* устойчивой по написанию;
* пригодной для переноса в `release_manifest.md` без переосмысления.

### Fifth) Проверить готовность к Stage 1

К концу Stage 0 должно быть понятно:

* какой комплект исходников ожидается;
* что Stage 1 будет работать уже с реальными файлами;
* что первый реальный freeze-паспорт появится именно на Stage 1;
* что последующая работа будет вестись через актуальный ZIP-архив репозитория после каждого завершённого шага.

---

## Canonical handoff to Stage 1

После Stage 0 переход выполняется в:

* `docs/authoring/pipeline/stage_1_freeze_sources/README.md`
* `docs/authoring/pipeline/stage_1_freeze_sources/message_prompt.md`

Stage 1 — первая рабочая стадия, на которой:

* раскладываются реальные инженерные исходники;
* появляется `release_manifest.md`;
* появляется стартовый `file_map.md`.

---

## Outputs

Stage 0 не создаёт project-specific markdown-артефакты.

Ожидаемый выход Stage 0 — это подтверждённая готовность начать Stage 1 без дополнительной перепланировки репозитория.

---

## DoD

Stage 0 завершён, если:

* оператор прочитал `/README.md` и `docs/authoring/pipeline/PIPELINE.md`;
* структура репозитория проверена и признана рабочей;
* канонические пути и границы зон понятны;
* рабочий Board ID определён;
* ясно, что первый канонический проектный артефакт будет создан на Stage 1;
* ясно, что дальнейшая работа идёт по актуальному ZIP-архиву репозитория.

---

## Common mistakes

* Пытаться выполнять freeze уже на Stage 0.
* Думать, что Stage 0 должен создавать `release_manifest.md` или `file_map.md`.
* Повторно расписывать весь layout репозитория вместо проверки, что он уже существует.
* Считать `docs/authoring/IMG/` допустимым путём.
* Не выбрать Board ID до перехода в Stage 1.
