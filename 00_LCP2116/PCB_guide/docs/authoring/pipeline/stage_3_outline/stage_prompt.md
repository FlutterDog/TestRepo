# STAGE PROMPT — Stage 3 (Outline)

Ты работаешь по канону репозитория:

* `docs/authoring/pipeline/PIPELINE.md`
* `/README.md`

Сейчас активная стадия: **Stage 3 — Outline**.

## GOAL

Собрать Stage three как **двухходовую интерактивную стадию**, а не как один файл с оглавлением.

Твой обязательный комплект по умолчанию:

* `outline.md`
* `author_notes.md`
* при необходимости `figure_brief_template.md`

`outline.md` должен заранее требовать для каждой крупной функциональной главы:

* назначение;
* аппаратную реализацию;
* ключевые характеристики;
* возможности и сценарии применения;
* ограничения и последствия модификации.

`author_notes.md` должен быть generated-by-assistant русским опросником по тем же главам.
Он должен быть project-specific: не копией общего шаблона, а развёрткой под фактические разделы текущего outline и planned assets.

---

## HARD RULES

### Один. Не останавливайся на одном `outline.md`

Stage three обязан вернуть не только `outline.md`, но и `author_notes.md`.

### Два. Каждая глава должна быть исполнимой по фактам

### Три. Каждая функциональная глава обязана иметь explanatory structure

### Три А. Outline обязан быть с нумерацией и нормальной вёрсткой

`outline.md` не может быть просто списком заголовков без структуры.

Обязательные требования:

* в начале файла должно быть нумерованное оглавление;
* использовать арабскую нумерацию разделов: `1.`, `1.1.`, `1.1.1.`;
* номера в оглавлении и в теле документа должны совпадать;
* крупные разделы должны разделяться `---`;
* финальный outline должен быть визуально читаемым и пригодным для ручного review.

Недопустимо:

* выдавать outline без нумерации;
* выдавать outline как рыхлый перечень заголовков;
* смешивать разные схемы нумерации внутри одного документа.

### Четыре. Figures Plan обязателен

Для каждой planned figure / table asset нужно:

* объяснить, зачем она нужна;
* указать источник;
* указать, можно ли её извлечь автоматически;
* указать, кто отвечает за подготовку;
* указать текущий статус.

### Пять. Если есть author-owned или mixed assets, выдай `figure_brief_template.md`

### Шесть. Ассистент обязан вести пользователя

После выдачи Stage three ассистент обязан явно сообщить:

* какие главы автору нужно заполнить в `author_notes.md`;
* какие figure-assets или tables должен подготовить автор;
* какие official references стоит догрузить, если их не хватает;
* какие style exemplars полезно догрузить, если они улучшат Figures Plan или Draft composition;
* можно ли уже идти на Stage four или нужен author pass.

### Семь. Не заставляй автора самому придумывать структуру notes-файла

Структура `author_notes.md` должна генерироваться ассистентом на основе текущего outline.

### Восемь. Не выдавай сырую копию шаблона

Если `author_notes_template.md` существует, используй его только как каноническую основу.

Финальный `author_notes.md` для проекта должен:

* повторять реальные заголовки и смысловые блоки текущего `outline.md`;
* адаптировать вопросы под тип главы;
* включать только релевантные для этой главы блоки;
* не содержать пустых универсальных секций, которые не относятся к проекту.

## OUTPUT FORMAT

Выдай:

* `## docs/authoring/artifacts/stage_3_outline/outline.md`
* `## docs/authoring/artifacts/stage_3_outline/author_notes.md`

Если есть author-owned / mixed figure-assets или missing assets, дополнительно выдай:

* `## docs/authoring/artifacts/stage_3_outline/figure_brief_template.md`

После файлов обязательно дай короткий handoff следующего шага:

* `Stage status`
* `Author action now`
* `Where to put it`
* `Next re-entry point`
* `Go / No-Go`

## QUALITY BAR

* Outline покрывает все ключевые подсистемы.
* Outline имеет нумерованное оглавление и совпадающую нумерацию в теле документа.
* Для каждой крупной главы заранее предусмотрен полезный, а не сухой способ раскрытия.
* Figures Plan поддерживает будущий живой текст.
* Для figures и tables понятна ответственность и текущий статус.
* `author_notes.md` уже готов к заполнению автором.
* После ответа ассистента пользователь не должен гадать, что делать дальше.

### Девять. Style exemplars допустимы, но не как factual source

TI / ADI / BeagleBoard manuals можно использовать как ориентир для того,
какие diagrams, tables и explanatory inserts уместны в document design.

Каноническое место таких документов:

* `docs/authoring/pipeline/style_references/`

Если они доступны, ты обязан использовать их только для:

* выбора плотности текста;
* выбора типов figures и tables;
* выбора мест, где лучше diagram / excerpt / table вместо длинного prose;
* выбора document rhythm между разделами.

Но они не могут подменять факты по вашей плате и не должны использоваться как proof-source.
