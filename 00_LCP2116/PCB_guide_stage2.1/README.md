# PCB Board Guide — репозиторий для выпуска Board Guide / System Reference Guide по печатной плате

Этот репозиторий — **производственный каркас** для подготовки и выпуска инженерного документа по плате: от замороженного комплекта исходников до финального релизного `board_guide.md`.

Процесс оптимизирован под сценарий **one-shot delivery**:

* плату проектировал автор документа;
* инженерные исходники доступны;
* документ пишет один человек или малая группа;
* сопровождение процесса документирования не считается бесконечным.


---

## ноль. Как взаимодействовать с ассистентом

Для запуска нового диалога, повторного входа в стадию и восстановления после потери контекста использовать:

* `docs/authoring/pipeline/START_HERE.md`
* `docs/authoring/pipeline/HELP.md`
* `docs/authoring/pipeline/REENTRY_PROMPTS.md`
* `docs/authoring/pipeline/style_references/README.md`

Это обязательная часть станка: автор не должен держать сценарий стадий в памяти.


Если нужен самый короткий вход без чтения длинных документов, автор может:

* загрузить актуальный архив репозитория;
* написать в чат `help`;
* получить краткую карту действий, стадий и готовых команд.

## один. Что является выходным продуктом

Основной результат работы репозитория — **финальный инженерный документ**:

* `docs/release/board_guide.md`

При наличии иллюстраций он сопровождается папкой:

* `docs/release/IMG/`

Целевой стиль выходного документа:

* **плотный инженерный текст**;
* описание платы как платформы и как конкретной реализованной аппаратуры одновременно;
* без process-language, без служебных статусов, без внутренних оговорок пайплайна;
* по тону — **между сухим board guide и system reference manual**.

Релизный документ должен отвечать на практические вопросы инженера, интегратора и нового владельца:

* что делает плата и где применяется;
* как устроены вычислитель, память, линии питания и интерфейсы;
* какие возможности открывает архитектура платы;
* какие параметры важны для интеграции и модификации;
* какие узлы критичны для повторения, bring-up, диагностики и контролируемых изменений.

## два. Scope / Boundaries

### два точка один. Что входит

Board Guide описывает только аппаратную часть:

* назначение платы и контекст применения;
* функциональную архитектуру;
* архитектуру питания и защиты;
* процессорный модуль, память и внешние контроллеры;
* интерфейсы и внешние соединения;
* ключевые характеристики узлов и подсистем;
* поддерживаемые аппаратные возможности платы;
* PCB / layout notes и manufacturing-аспекты, если они подтверждены источниками;
* базовые аппаратные проверки и bring-up на уровне железа.

### два точка два. Что не входит

Документ **не является**:

* руководством по эксплуатации;
* описанием прошивки и протоколов на уровне кода;
* полным даташитом на каждую микросхему;
* маркетинговым буклетом.

В релизный документ не попадают:

* `FACT / ASSUMPTION / UNKNOWN` как служебные маркеры;
* open items;
* внутренние комментарии о том, как писался текст;
* `TBD` / заглушки / обещания «уточнить позже».

## три. Source of Truth и правило инженерной честности

Репозиторий строится вокруг **одного замороженного комплекта исходников** в `sources/`.

Но для качественного инженерного текста различаются **три класса источников**:

### три точка один. Board implementation sources

Подтверждают, **что реально реализовано на данной плате**:

* schematic;
* native CAD;
* PCB exports;
* BOM;
* Gerber / drill;
* stackup;
* board-specific manuals в `sources/extras/`.

### три точка два. Component and module reference sources

Подтверждают, **что умеют применённые микросхемы, модули и контроллеры**:

* официальные datasheet;
* reference manual;
* product brief;
* официальные module manuals;
* официальные application notes — только если они реально нужны для базовых характеристик.

Такие материалы должны быть либо физически помещены в `sources/extras/references/`, либо явно перечислены в `reference_sources.md`.

### три точка два точка один. Style exemplars

Дополнительно можно использовать style-only документы-образцы:

* TI board guide;
* ADI system reference manual;
* Beagle style manual;
* другие хорошо собранные board / system reference guides.

Их место в репозитории:

* `docs/authoring/pipeline/style_references/`

Они используются только как ориентир по:

* композиции;
* плотности текста;
* набору figures и tables;
* расположению diagrams и schematic excerpts.

Они не являются source of truth по вашей плате и не могут заменять board facts или component facts.

### три точка три. Author intent / design rationale

Подтверждает, **зачем узел введён и какую роль он играет в изделии**:

* комментарии автора;
* design intent;
* chapter notes по outline;
* инженерные пояснения по сценариям применения.

Это не заменяет board facts, но позволяет писать документ не как список компонентов, а как объяснение архитектуры.

### три точка четыре. Главное правило качества

Любое значимое утверждение должно быть либо:

* подтверждено источником нужного класса;
* либо остаться в authoring-артефактах как open item;
* либо не войти в релиз вообще.

Канонические сущности процесса:

* `release_manifest.md` — паспорт Source of Truth;
* `file_map.md` — трассируемость ключевых утверждений;
* `reference_sources.md` — разрешённые внешние официальные источники по компонентам и модулям;
* `design_intent.md` — rationale и роль узлов;
* `author_notes.md` — chapter-aligned заметки автора для живого текста;
* `open_items.md` — всё неподтверждённое, что не должно просочиться в релиз.

## четыре. Release vs Authoring

### четыре точка один. Release

`docs/release/` — то, что отдаётся наружу.

В финальном состоянии проекта здесь находятся:

* `board_guide.md`;
* `IMG/` — только реально используемые изображения.

Release должен быть:

* самодостаточным;
* читабельным для инженера;
* достаточно объясняющим, чтобы новый владелец понял плату как платформу;
* очищенным от process-language и служебных артефактов.

### четыре точка два. Authoring

`docs/authoring/` — внутренняя рабочая зона.

Здесь живут:

* артефакты стадий;
* трассируемость;
* glossary;
* open items;
* reference sources;
* design intent;
* chapter notes;
* draft-версии;
* qc-report;
* pipeline-регламент.

Authoring может быть честно шероховатым. Release — нет.

   │     ├─ IMG/

### Важное правило по изображениям

**Единственная каноническая папка authoring-изображений:**

* `docs/authoring/artifacts/IMG/`

Именно из неё изображения попадают в:

* `docs/release/IMG/`

Если в старых текстах встречается `docs/authoring/IMG/`, считать это устаревшим путём и не использовать.

## шесть. Как начать работу

Порядок старта:

* клонировать репозиторий или создать новый репозиторий из template;
* прочитать `/README.md`;
* перейти к `docs/authoring/pipeline/PIPELINE.md`;
* выполнить Stage zero и проверить готовность репозитория к реальной работе;
* только после этого загрузить и разложить freeze-комплект исходников в `sources/` в рамках Stage one;
* далее выполнять стадии последовательно.

Отдельных скриптов инициализации структуры не требуется: структура поставляется вместе с репозиторием.

### Рабочий цикл по архиву репозитория

Этот template рассчитан на такой режим работы с ассистентом:

* пользователь загружает актуальный ZIP-архив репозитория;
* ассистент выполняет только активную стадию внутри этого архива;
* результаты сохраняются в том же репозитории;
* обновлённый репозиторий снова архивируется и используется как Source Bundle для следующей стадии.

### Отдельное правило по инженерным заметкам автора

После появления `outline.md` автор **может и должен** при желании добавить chapter-aligned заметки в:

* `docs/authoring/artifacts/stage_3_outline/author_notes.md`

Этот файл не входит в релиз, но помогает:

* зафиксировать design intent по главам;
* объяснить, что клиент должен понять из каждой главы;
* добавить живой инженерный смысл без потери дисциплины по фактам.

## семь. Где описан процесс

Канон процесса ведётся в:

* `docs/authoring/pipeline/PIPELINE.md`

Детальные инструкции по стадиям лежат в:

* `docs/authoring/pipeline/stage_*/README.md`

Если `README` стадии и `stage_prompt.md` расходятся:

* `stage_prompt.md` имеет приоритет как рабочая инструкция для генерации;
* `README` стадии остаётся поясняющим документом процесса.

## восемь. Ожидаемое качество выходного документа

Финальный `docs/release/board_guide.md` должен быть:

* инженерным, а не канцелярским;
* содержательным, а не машинным списком;
* честным по фактам, но без самооправдательных оговорок;
* близким по ощущению к **среднему режиму между board guide и system reference manual**.

Практический признак хорошего релиза:

* инженер и интегратор читают документ как описание платы,
* понимают, что на ней реализовано,
* какие характеристики и ограничения критичны,
* и что эта архитектура позволяет делать в реальном изделии.

## Практическая модель работы: кто, когда и что делает

### Роль ассистента

Ассистент:

* читает freeze-комплект и артефакты предыдущих стадий;
* предлагает структуру документа, перечень рисунков, схем и таблиц;
* указывает, какие иллюстрации уже можно собрать из имеющихся материалов, а какие нужно подготовить отдельно;
* пишет `outline.md`, `fact_pack.md`, `draft_v0.md` и другие stage-артефакты;
* ведёт пользователя по стадиям и явно говорит, какой следующий шаг нужен от автора.

### Роль автора платы / владельца проекта

Автор:

* замораживает исходники и reference-материалы;
* при необходимости добавляет официальные PDF / datasheet в `sources/extras/references/`;
* проверяет `open_items.md`;
* после утверждения `outline.md` при желании заполняет `author_notes.md`;
* по необходимости готовит и загружает иллюстрации, схемы, таблицы и другие figure-assets, если их нельзя корректно извлечь автоматически.

### Где обычно требуется участие автора

Наиболее полезные точки участия автора:

* конец Stage two — подтвердить спорные места и design intent;
* середина Stage three — утвердить Figures Plan и решить, какие figure-assets надо подготовить;
* после утверждения outline — при необходимости заполнить `author_notes.md` по главам;
* до конца Stage four — догрузить недостающие официальные reference-docs и / или figure-assets.

## шесть. Режим работы с автором

Этот репозиторий предполагает **assistant-led workflow**.

То есть ассистент должен не только создавать артефакты стадий, но и в конце каждой стадии явно вести автора дальше:

* что уже сделано;
* что теперь нужно от автора;
* какие файлы надо подготовить или догрузить;
* куда именно их положить;
* что вернуть ассистенту для продолжения.

Особенно это важно на `Stage three`, где ассистент обязан подготовить:

* `outline.md`;
* `author_notes.md`;
* при необходимости `figure_brief_template.md`.

Автор заполняет только уже подготовленные файлы, а не придумывает структуру взаимодействия с нуля.
