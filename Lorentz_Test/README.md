# Lorentz Test

**Автоматизированная утилита и стенд проверки контроллеров и модулей Lorentz**

Статус документа: **проект плана**  
Поддерживаемая базовая прошивка: **LCP Basic Firmware v1.02.0**

---

## 1. Назначение проекта

Проект `Lorentz_Test` предназначен для автоматизированной производственной и сервисной проверки:

- центрального контроллера LCP2116;
- модулей LDO;
- модулей LDI;
- последующего добавления LAI, LCT и других модулей Lorentz.

Базовая прошивка LCP `v1.02.0` считается завершённой и не изменяется для реализации стенда. Проверка строится вокруг уже доступных интерфейсов и функций прошивки.

---

## 2. Основной принцип архитектуры

Функции распределяются следующим образом:

- **LCP Basic Firmware v1.02.0** выполняет аппаратные операции, Modbus RTU, Modbus TCP, X2X, диагностику и хранение конфигурации;
- **Python backend** управляет последовательностью тестов, интерфейсами, проверками PASS/FAIL и формированием протокола;
- **LabVIEW frontend** предоставляет операторский интерфейс, отображает ход проверки и итоговые результаты;
- **испытательный стенд** формирует физические подключения и внешние ответы, необходимые для проверки интерфейсов и модулей.

В базовую прошивку не добавляются bridge-режим, отдельная команда `test start` или специальная логика ОТК.

---

## 3. Предлагаемый состав стенда LCP

### 3.1 Подключение контроллера

- USB Service CDC контроллера подключается непосредственно к компьютеру;
- ETH1 и ETH2 подключаются к общему Ethernet-коммутатору;
- компьютер подключается к тому же коммутатору;
- S1, S2, S3, S4 и HMI подключаются к пяти независимым каналам RS-485;
- в качестве многоканального преобразователя предлагается использовать Moxa или эквивалентный serial device server;
- microSD устанавливается в контроллер для проверки детектирования, инициализации и файловых операций;
- X2X подключается к тестируемым модулям.

На стенде одновременно проверяется только один LCP, поскольку заводские IP-адреса у разных контроллеров совпадают.

### 3.2 RS-485 S1–S4

Для каждого канала компьютер через Moxa запускает независимый Modbus RTU slave с адресом 1.

LCP выполняет штатный запрос:

- функция 0x03;
- начальный регистр 0;
- количество регистров 2;
- период опроса 300 мс.

Каждый канал возвращает уникальную пару значений. Это позволяет однозначно проверить соответствие физического порта и опубликованных данных.

Пример тестовых значений:

- S1: 0x1101, 0x1102;
- S2: 0x2201, 0x2202;
- S3: 0x3301, 0x3302;
- S4: 0x4401, 0x4402.

Проверяется полный тракт:

LCP UART → RS-485 передатчик → Moxa → программный Modbus slave → Moxa → RS-485 приёмник → LCP Modbus RTU → внутренние данные → Modbus TCP → компьютер.

### 3.3 Порт HMI

В прошивке `v1.02.0` HMI работает в режиме echo-test.

Компьютер через отдельный канал Moxa передаёт несколько тестовых последовательностей и проверяет точное совпадение принятого ответа:

- короткая фиксированная последовательность;
- последовательность с 0x00 и 0xFF;
- пакет увеличенной длины;
- серия повторных пакетов.

### 3.4 Ethernet ETH1 и ETH2

Оба Ethernet-порта подключаются к одному коммутатору и проверяются независимо.

Для каждого порта выполняются:

- проверка физического link;
- открытие TCP-порта 502;
- чтение Modbus TCP holding registers;
- сравнение данных S1–S4 с тестовыми значениями;
- проверка quality-регистров;
- серия повторных запросов;
- контроль transport errors и malformed requests.

Проверка выполняется по штатным адресам тестового профиля. Компьютеру назначается адрес той же подсети.

## 4. Проверка LCP

### 4.1 Идентификация и конфигурация

1. Найти устройство USB по VID/PID.
2. Проверить USB configuration protocol.
3. Прочитать версию прошивки.
4. Допустить к тесту только `LCP Basic Firmware 1.02.0`.
5. Сохранить исходную конфигурацию.
6. Записать `factory_test_config.json` через бинарный USB-протокол.
7. Выполнить reboot и дождаться повторного подключения.
8. Прочитать конфигурацию обратно и проверить CRC32 и полное совпадение bundle.

### 4.2 Диагностика контроллера

Через текстовую USB-консоль считываются:

- `version`;
- `rtos`;
- `config status`;
- `field`;
- `x2x`;
- `eth`;
- `sd`;
- `battery`;
- `time` и `rtc`;
- `watchdog`;
- `rs485`;
- `sc16is`.

Парсер привязывается к формату вывода прошивки `v1.02.0` и извлекает именованные поля `key = value`. Порядок строк и количество пробелов не используются как часть логики.

### 4.3 Начальные критерии PASS/FAIL для LCP

- USB определяется и отвечает;
- версия прошивки соответствует поддерживаемой;
- запись и чтение конфигурации проходят без ошибки;
- CRC32 после записи совпадает;
- FreeRTOS scheduler работает;
- heap и stack находятся в допустимых пределах;
- S1–S4 получают ожидаемые пары регистров;
- UART errors равны нулю;
- ETH1 и ETH2 доступны по Modbus TCP;
- данные S1–S4 одинаково корректно читаются через ETH1 и ETH2;
- HMI возвращает переданные последовательности без искажений;
- microSD определяется и проходит файловый тест;
- RTC читается;
- цепь резервной батареи имеет корректное состояние;
- watchdog включён и имеет ожидаемую конфигурацию;
- X2X CRC self-test проходит.

## 5. Проверка LDO и LDI

### 5.1 Предлагаемая схема

LDO и LDI подключаются к X2X контроллера. Контакты каждого реле LDO соединяются с соответствующими входами LDI через испытательный жгут.

Предпочтительный вариант проверяет оба состояния контактов:

- нормально открытый контакт;
- нормально закрытый контакт.

При восьми реле требуется до шестнадцати дискретных входов.

### 5.2 Последовательность проверки одного реле

1. Проверить связь с LDO и LDI.
2. Установить реле в OFF.
3. Проверить ожидаемое состояние NO и NC на LDI.
4. Установить реле в ON.
5. Проверить ожидаемое состояние NO и NC на LDI.
6. Повторить переключение несколько раз.
7. Зафиксировать счётчики ошибок и итог канала.

Так проверяются одновременно:

- X2X;
- адресация модулей;
- драйвер выхода LDO;
- катушка реле;
- NO- и NC-контакты;
- входной канал LDI;
- обратное чтение результата контроллером.

---

## 6. Протокол испытаний

Для каждого запуска сохраняются:

- серийный номер контроллера или модуля;
- дата и время;
- оператор;
- версия тестовой утилиты;
- версия тестового профиля;
- версия прошивки LCP;
- исходная конфигурация;
- применённая тестовая конфигурация;
- результаты отдельных этапов;
- измеренные значения и счётчики;
- диагностический вывод;
- общий результат PASS или FAIL.

Основные форматы:

- JSON — полный машинный результат;
- TXT — читаемый производственный протокол;
- PDF может формироваться операторской программой позднее.

---

## 7. Этапы разработки

### Этап 1. Зафиксировать требования и стенд LCP

- определить точную модель Moxa;
- определить режим подключения к её каналам;
- подготовить схему подключения ETH1, ETH2, S1–S4 и HMI;
- определить тестовые IP-адреса;
- утвердить критерии PASS/FAIL.

### Этап 2. Python backend LCP

- USB discovery;
- работа с binary configuration protocol;
- управление текстовой консолью;
- программные Modbus RTU slaves для S1–S4;
- Modbus TCP client для ETH1/ETH2;
- parser диагностических ответов `v1.02.0`;
- формирование JSON/TXT отчёта.

### Этап 3. Проверка LDO и LDI

- схема испытательного жгута;
- карта соответствия реле и входов;
- алгоритм OFF/ON;
- критерии NO/NC;
- протокол по каждому каналу.

### Этап 4. LabVIEW frontend

- запуск теста;
- ввод серийного номера и оператора;
- отображение этапов;
- отображение PASS/FAIL;
- просмотр подробностей ошибки;
- сохранение и печать протокола.

### Этап 5. Расширение

- LAI;
- LCT;
- другие модули Lorentz.

---

## 8. Открытые вопросы

Перед реализацией необходимо уточнить:

1. Точная модель Moxa и количество доступных независимых RS-485 каналов.
2. Используется режим Virtual COM, TCP Server/Client или другой режим Moxa.
3. Может ли один компьютер одновременно открыть пять независимых каналов.
4. Какой источник питания и уровни сигналов используются в испытательном жгуте LDO–LDI.
5. Проверяются только NO-контакты или одновременно NO и NC.
6. Где находится текущий исходный проект LabVIEW и можно ли использовать его интерфейс как основу.
7. Какие поля должны входить в официальный производственный протокол.

---

## 9. Зафиксированные ограничения

- LCP Basic Firmware `v1.02.0` не изменяется;
- USB–RS-485 bridge не разрабатывается;
- отдельная команда `test start` в прошивку не добавляется;
- автоматическая проверка строится на существующих USB, Modbus RTU, Modbus TCP, X2X и диагностических интерфейсах;
- тестовая утилита является отдельным проектом и не входит в базовую прошивку LCP.
