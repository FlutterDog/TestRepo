# 1. Описание создаваемого Объекта

Модуль учета коммутационного ресурса **LCT1114 Revision 3** должен представлять собой специализированный модуль линейки Lorentz, предназначенный для мониторинга параметров работы высоковольтных выключателей и учета их механического, электрического и коммутационного ресурса.

Модуль должен быть выполнен в общем форм-факторе модулей линейки Lorentz, в пластиковом корпусе для установки на DIN-рейку.

Модуль должен быть разработан на базе процессорной платформы **ARM Cortex-M7**.

Модуль должен обеспечивать работу:

* в составе системы Lorentz совместно с процессорным модулем IGAS LCP / LCP2116;
* в самостоятельном режиме с обменом данными по Modbus RTU и Modbus TCP;
* в сервисном режиме через интерфейс USB.

Модуль должен обеспечивать локальную регистрацию, обработку и хранение данных коммутационных событий с возможностью последующей передачи данных в головной модуль LCP, систему верхнего уровня или конфигурационную утилиту LCT Toolbox.

Модуль должен быть адресным устройством и обеспечивать возможность назначения уникального сетевого адреса.

В составе модуля должны быть предусмотрены:

* входы контроля фазных токов;
* два измерительных диапазона для каждого фазного тока;
* входы контроля конечных положений выключателя;
* три входа контроля токов электромагнитов включения и отключения;
* вход датчика температуры;
* оперативная память для регистрации коммутационных событий;
* энергонезависимая память для локального хранения событий;
* часы реального времени;
* интерфейсы RS-485 / X2X, Ethernet и USB.

# 2. Практическое назначение Объекта

Модуль LCT1114 Revision 3 предназначен для применения в составе систем мониторинга, диагностики и автоматизации высоковольтных выключателей.

Модуль должен обеспечивать:

* контроль операций включения и отключения;
* контроль конечных положений выключателя;
* измерение фазных токов при коммутациях;
* регистрацию осциллограмм фазных токов;
* регистрацию осциллограмм токов электромагнитов включения и отключения;
* определение максимальных и расчетных значений токов электромагнитов;
* определение времени горения дуги;
* расчет времени операций включения и отключения;
* расчет разновременности фаз;
* учет количества операций;
* учет механического ресурса;
* учет электрического и коммутационного ресурса;
* диагностику неполнофазных и незавершенных операций;
* локальное хранение коммутационных событий;
* передачу измеренных, расчетных, диагностических и осциллографических данных;
* формирование файлов COMTRADE средствами конфигурационной утилиты LCT Toolbox.

Модуль должен применяться для контроля технического состояния высоковольтных выключателей, анализа коммутационных процессов и планирования технического обслуживания.

# 4. Функциональные требования

## 4.1. Контроль фазных токов и коммутационного процесса

Модуль должен обеспечивать подключение трех входов контроля фазных токов выключателя.

Для каждого фазного тока должны быть предусмотрены два измерительных диапазона:

* High Gain — для измерения токов малого и среднего уровня;
* Low Gain — для измерения токов большого уровня без насыщения измерительного тракта.

Во время регистрации активного события данные должны приниматься по обоим измерительным диапазонам.

Во время регистрации допускается временное сохранение данных всех шести измерительных трактов фазного тока в оперативной памяти.

После завершения события встроенное программное обеспечение должно выбирать корректный измерительный диапазон отдельно для каждой фазы.

В окончательном событии должна сохраняться одна выбранная осциллограмма для каждой фазы. Завершенное событие должно содержать три осциллограммы фазных токов.

Долговременное хранение одновременно двух измерительных диапазонов одной фазы не требуется.

Модуль должен обеспечивать:

* регистрацию формы фазных токов;
* определение действующего значения тока;
* определение тока последнего отключения;
* определение времени горения дуги;
* расчет электрического ресурса последней коммутации;
* расчет общего накопленного электрического ресурса;
* формирование предупредительных и аварийных статусов.

## 4.2. Контроль конечных положений выключателя

Модуль должен обеспечивать подключение шести дискретных сигналов контроля конечных положений выключателя.

Модуль должен обеспечивать:

* определение состояния выключателя по фазам;
* контроль состояний «включено» и «отключено»;
* выявление конфликтных состояний;
* выявление неполнофазного переключения;
* программное подавление дребезга;
* контроль недостижения целевого положения;
* ручной сервисный режим проверки входов.

## 4.3. Контроль токов электромагнитов включения и отключения

Модуль должен обеспечивать подключение трех выносных датчиков токов электромагнитов включения и отключения.

Модуль должен обеспечивать:

* измерение токов электромагнитов;
* регистрацию формы тока во время коммутации;
* определение максимального и расчетного значения тока;
* сохранение осциллограмм трех каналов токов электромагнитов;
* передачу осциллограмм и расчетных значений в LCT Toolbox, LCP или систему верхнего уровня.

Состав и назначение трех каналов должны определяться аппаратной конфигурацией, настройками и программной документацией.

## 4.4. Контроль температуры

Модуль должен обеспечивать подключение термосопротивления PT100.

Измеренная температура должна использоваться для:

* отображения;
* передачи в головной модуль;
* сохранения в составе события;
* применения в диагностических алгоритмах в объеме программной реализации.

## 4.5. Контроль времени операции

Модуль должен обеспечивать измерение времени операций включения и отключения выключателя.

Начало операции должно определяться по событию включения или отключения либо по устойчивому изменению соответствующего сигнала конечного положения.

Модуль должен обеспечивать расчет времени достижения целевого конечного положения отдельно по фазам.

Временные параметры должны сравниваться с заданными уставками.

По результатам сравнения должны формироваться статусы:

* норма;
* предупреждение;
* авария.

## 4.6. Контроль разновременности фаз

Модуль должен обеспечивать расчет разновременности фаз при включении и отключении.

Разновременность должна определяться как разница между максимальным и минимальным временем достижения целевого положения по фазам.

Рассчитанное значение должно сохраняться в составе события и передаваться в головной модуль или систему верхнего уровня.

## 4.7. Диагностика незавершенных операций и механических аномалий

Модуль должен обеспечивать выявление следующих состояний:

* задержка операции включения или отключения;
* недостижение целевого конечного положения;
* незавершенная операция;
* неполнофазное переключение;
* конфликтное состояние конечных положений;
* нестабильность или дребезг входных сигналов;
* превышение предупредительных или аварийных временных уставок.

## 4.8. Расчет механического ресурса

Модуль должен обеспечивать учет количества операций включения и отключения.

Учет должен выполняться:

* по фазам;
* агрегированно для выключателя;
* отдельно для включений и отключений в объеме программной реализации.

Механический ресурс должен рассчитываться на основании накопленного количества операций и заданного паспортного ресурса выключателя.

Счетчики и параметры ресурса должны сохраняться в энергонезависимой памяти.

## 4.9. Учет электрического и коммутационного ресурса

Модуль должен обеспечивать:

* расчет электрического ресурса каждой коммутации;
* расчет накопленного электрического ресурса по фазам;
* расчет израсходованного электрического ресурса в процентах;
* учет коммутаций токов короткого замыкания;
* формирование предупредительных и аварийных статусов;
* сохранение расчетных параметров в составе события.

Формулы расчета должны определяться программной документацией.

## 4.10. Регистрация коммутационных событий

Каждое завершенное коммутационное событие должно содержать:

* уникальный идентификатор события;
* дату и время;
* тип операции;
* состояние конечных положений;
* три выбранные осциллограммы фазных токов;
* три осциллограммы токов электромагнитов;
* период дискретизации;
* количество точек;
* расчетные временные параметры;
* время горения дуги;
* максимальные и расчетные значения токов;
* значения механического и электрического ресурса;
* диагностические статусы;
* признаки целостности и завершенности записи.

Модуль должен обеспечивать кольцевую регистрацию не менее трех последовательных коммутационных событий без смешивания данных разных событий.

Во время регистрации активного события допускается временное хранение данных шести измерительных трактов фазного тока.

После завершения обработки события должны сохраняться три выбранных фазных канала и три канала токов электромагнитов.

## 4.11. Локальное хранение событий

Модуль должен обеспечивать запись завершенных коммутационных событий в энергонезависимую Flash-память.

Локальный архив должен обеспечивать хранение не менее 100 завершенных коммутационных событий.

Хранение должно быть организовано таким образом, чтобы повреждение или незавершенная запись одного события не приводили к повреждению ранее сохраненного архива.

Для каждого события должны быть предусмотрены:

* идентификатор;
* версия формата;
* длина данных;
* контрольная сумма;
* состояние записи;
* дата и время.

Запись события должна выполняться с фиксацией признака завершенности после успешной записи данных и проверки их целостности.

При заполнении архива должна быть предусмотрена циклическая перезапись наиболее старых событий либо иной механизм управления архивом, определенный программной документацией.

## 4.12. Передача данных и взаимодействие с системой

Модуль должен обеспечивать передачу измеренных, расчетных, диагностических и осциллографических данных:

* через X2X;
* через Modbus RTU;
* через Modbus TCP;
* через сервисный интерфейс USB.

Карта регистров и структура команд должны быть описаны в программной документации.

Для осциллографических данных должен быть предусмотрен отдельный механизм чтения событий с контролем:

* идентификатора события;
* номера блока данных;
* количества точек;
* состояния готовности;
* целостности считанных данных.

Прерывание чтения не должно приводить к повреждению сохраненного события.

Повторное чтение одного и того же события должно возвращать одинаковый набор данных до момента удаления или перезаписи события.

## 4.13. Формирование COMTRADE

Модуль должен сохранять структурированные данные, необходимые для формирования файлов COMTRADE.

Формирование файлов COMTRADE должно выполняться конфигурационной утилитой LCT Toolbox.

Файл COMTRADE должен содержать:

* три осциллограммы фазных токов;
* три осциллограммы токов электромагнитов;
* дискретные состояния конечных положений;
* частоту и период дискретизации;
* временную метку;
* тип операции;
* наименования и единицы измерения каналов;
* масштабные коэффициенты;
* диагностические параметры в объеме программной реализации.

## 4.14. LCT Toolbox

Должна быть выполнена адаптация конфигурационной утилиты LCT Toolbox для работы с LCT1114 Revision 3.

LCT Toolbox должен обеспечивать:

* подключение по Modbus RTU;
* подключение по Modbus TCP;
* подключение через USB;
* выбор адреса или IP-адреса устройства;
* чтение и запись настроек;
* чтение и запись калибровочных параметров;
* чтение списка сохраненных событий;
* загрузку выбранного события;
* отображение осциллограмм;
* отображение расчетных и диагностических параметров;
* экспорт данных в CSV;
* экспорт данных в COMTRADE;
* проверку полноты и целостности считанных данных;
* отображение ошибок обмена и состояния устройства.

## 4.15. Калибровка

Модуль должен обеспечивать хранение и изменение калибровочных коэффициентов измерительных каналов.

Должны быть предусмотрены:

* коэффициенты усиления;
* смещения нуля;
* калибровочные точки;
* параметры измерительных диапазонов;
* коэффициенты каналов фазных токов;
* коэффициенты каналов токов электромагнитов;
* параметры температурного канала.

Запись калибровочных данных должна выполняться через LCT Toolbox или предусмотренный сервисный интерфейс.

## 4.16. Индикация и диагностика состояния модуля

Модуль должен иметь светодиодную индикацию:

* наличия питания;
* состояния связи;
* регистрации события;
* записи данных;
* исправности модуля.

Модуль должен обеспечивать диагностику:

* состояния процессора;
* состояния оперативной памяти;
* состояния Flash-памяти;
* состояния часов реального времени;
* состояния интерфейсов;
* количества сохраненных событий;
* ошибок записи и чтения данных;
* причин последнего перезапуска.

# 5. Выбор технологий реализации

Встроенное программное обеспечение должно быть разработано для процессорной платформы ARM Cortex-M7.

Конкретная модель процессора, тип АЦП, тип энергонезависимой памяти, Ethernet PHY или контроллер и состав интерфейсных компонентов определяются Исполнителем при разработке принципиальной схемы.

Разработка должна выполняться с использованием модульной структуры программного обеспечения.

В программном обеспечении должны быть выделены:

* аппаратный уровень;
* драйверы периферии;
* измерительный уровень;
* Event Recorder;
* система хранения событий;
* расчетные алгоритмы;
* коммуникационный уровень;
* сервисный уровень;
* интерфейс взаимодействия с LCT Toolbox.

Встроенное программное обеспечение должно обеспечивать разделение измерительных, архивных, расчетных, коммуникационных и сервисных задач.

Для реализации указанного разделения допускается применение операционной системы реального времени RTOS либо эквивалентной программной архитектуры.

Существующие алгоритмы LCT Revision 2 должны быть перенесены и адаптированы к новой аппаратной платформе.

Подробные формулы, внутренние переменные, структуры памяти, карта регистров, тестовые сценарии и алгоритмы обработки должны быть приведены в программной и сервисной документации.

При выборе процессорной платформы и сетевой подсистемы должна учитываться возможность дальнейшего расширения изделия, включая:

* использование одного оптического интерфейса Ethernet 100BASE-FX вместо базового интерфейса 100BASE-TX;
* реализацию обмена по МЭК 61850 MMS и GOOSE;
* подключение дополнительных быстродействующих, счетных или энкодерных входов;
* расширение функций диагностики и анализа коммутационных процессов.

Указанные функции не входят в текущий объем разработки LCT1114 Revision 3.

Разработка информационной модели МЭК 61850, файлов SCL / ICD / CID, протокольного стека МЭК 61850, интерфейса 100BASE-FX, PRP, а также проведение испытаний на соответствие требованиям МЭК 61850 должны согласовываться и оплачиваться отдельно.

Поддержка PRP в текущей аппаратной конфигурации не предусматривается.

# 6. Описание этапов разработки

В состав этапа входят:

* разработка структурной схемы;
* разработка принципиальной схемы;

* подготовка конструкторской документации.

Для выполнения аппаратной и программной отладки Исполнителю должен быть доступен как минимум один собранный опытный образец.

Стоимость изготовления печатных плат, комплектующих, сборки и доставки опытных образцов в стоимость разработки не входит и согласовывается отдельно.

Опытные образцы могут быть изготовлены:

* по заказу Исполнителя с последующей компенсацией Заказчиком фактических расходов на изготовление, комплектующие, сборку и доставку;

Рекомендуется изготовить не менее двух опытных образцов:

* один образец для проверки Заказчиком;
* один образец для отладки и сопровождения у Исполнителя.

## 6.4. Разработка и перенос встроенного программного обеспечения

В состав этапа входят:

* реализация Modbus RTU;
* реализация Modbus TCP;
* реализация USB Device;
* реализация часов реального времени;
* реализация сервисных функций;
* реализация диагностики.

## 6.5. Доработка LCT Toolbox

В состав этапа входят:

* поддержка новой карты данных;
* поддержка Modbus RTU;
* поддержка Modbus TCP;
* поддержка USB;
* чтение списка событий;
* загрузка и проверка событий;
* отображение осциллограмм;
* экспорт CSV;

* отображение диагностических и сервисных данных.

## 6.6. Первичная проверка опытного образца

Первичная проверка должна включать:

* проверку входного питания;
* проверку внутренних напряжений;
* запуск процессора;
* проверку программирования и отладки;
* проверку оперативной памяти;
* проверку Flash-памяти;
* проверку часов реального времени;
* проверку RS-485 / X2X;
* проверку Ethernet;
* проверку USB;
* проверку дискретных входов;
* проверку фазных измерительных каналов;
* проверку каналов токов электромагнитов;

Первичная проверка выполняется на доступном Исполнителю оборудовании и испытательных средствах.

* сборка и доставка повторных опытных образцов;
* неограниченное количество последующих аппаратных ревизий;
* изменение ранее согласованных технических требований.

Необходимость изготовления следующей аппаратной ревизии и проведения повторных испытаний определяется по результатам первичной проверки и согласовывается сторонами отдельно.
